# 知识卡片修正

通过 Prisma 直接修正知识卡片数据库记录，重点处理：

- `summary` / `body` 内容错误或过时
- `sourceExcerpt` 缺失、错摘、图片错配
- `sourceUrl` / `sourceDescription` / `sourceType` 填错
- 重复卡片合并、旧卡归档

## 前置约束

1. 默认在项目根目录内工作，直接读写当前项目数据库。
2. **不要新增来源中没有的实质性结论**。body 只能整理、排列、浓缩原文已有信息；新增判断必须来自 sourceExcerpt 或用户明确授权的外部信息。
3. **不要为了修卡默认启动 `npm run dev`**。用 Prisma 改库本身不需要长期运行服务；只有当用户明确要求你用后台页面复核展示效果时，才考虑启动开发服务。
4. 涉及以下动作时，先遵守用户授权规则：
   - 联网核对原网页、访问外部 API
   - 修改项目外文件
   - 启动长期运行服务
5. 修改前先定位卡片并向用户说明要改什么；修改后必须回查验证。

## 通用原则

### 拆分复杂原文

当一条 sourceExcerpt 内容复杂、覆盖多个子主题，需要拆分为多张卡片时：

1. **分类维度决定完整性**。不按原文叙述段落切，而按"读者需要知道什么"分。原文是叙事流（"先A再B然后C"），卡片不是——每张卡只聚焦一个子主题，把相关信息展开到最充分。
2. **原子化重组**。先将原文的知识点抽离成独立原子（如：申请材料、强com/弱com、超发offer机制、本校政策、跨专业策略、特殊招生类型……），再用这些原子拼成读者需要的主题卡片。卡片之间不保留原文的叙事顺序。
3. **横切知识点显式处理**。若有知识点横跨多个子主题（如"申请材料"同时适用于夏令营和预推免），不应只归给一张卡，而应在每张相关卡中都保留对应版本，或单独建卡。
4. **逐句覆盖校验**。分拆完毕后，逐句扫描原文 sourceExcerpt，确保每句话至少被一张目标卡的 body 覆盖。落空的句子就是遗漏信号，需要判断是补充到现有卡还是新建卡。

### 拆分 vs 合并的决策框架

面对一条或多条内容，先判断：

| 条件 | 操作 |
|------|------|
| 同一主题的不同角度的信息散在多张卡 | 合并 |
| 一条 sourceExcerpt 覆盖多个子主题 | 拆分 |
| 一张卡的 body 混入两个以上独立主题 | 拆分 |
| 多张卡的 body 内容高度重合 | 合并 |

拆分后必须做逐句覆盖校验。合并后必须确认无信息丢失。

## 推荐脚本

优先使用仓库内脚本快速审计图片归属：

```bash
node .agents/skills/fix-cards/scripts/audit-card-images.js --id <card-id>
node .agents/skills/fix-cards/scripts/audit-card-images.js --summary "<summary关键词>"
```

脚本会输出：
- 目标卡的基本信息
- `sourceExcerpt` 中引用的本地图片
- 每张图片被哪些卡引用
- 是否存在跨 `sourceUrl` 复用
- 同来源的兄弟卡列表

更新长文本字段时优先使用专用脚本，不要把整段中文正文直接拼进 `node -e`：

```powershell
$env:CARD_SUMMARY = @'
新的标题
'@
$env:CARD_BODY = @'
新的正文
'@
node .agents/skills/fix-cards/scripts/update-card-fields.js --id <card-id> --summary-env CARD_SUMMARY --body-env CARD_BODY --apply
```

脚本默认 dry-run，必须加 `--apply` 才会写库；它会做字段白名单检查并在写入后回查。

## 修正流程

### 正文组织优化：结构化、简单易读

当用户反馈正文"不够简洁""不够优美""太散""不易读""要结构化"时，优先把 `body` 改成可扫读结构，而不是继续扩写：

1. 用"结论先行 + 标签小节 + 短句/短列表"的结构，减少长段落。
2. 常用标签可按内容选择：【结论】、【一句话定位】、【适合谁】、【不适合谁】、【主要优势】、【主要风险】、【入学后建议】、【未来方向】。
3. 对"值不值得""适不适合""怎么选"类卡片，优先写清楚适合/不适合人群，再写优势和风险。
4. 正文应尽量短而完整：能用 500-800 字说清的卡，不写成多段散文；避免重复解释同一点。
5. 修改后验证 `body` 包含预期标签，来源字段未变，中文无乱码。

### Step 1: 解析用户输入

识别修正模式：

| 模式 | 示例 | 处理方式 |
|------|------|----------|
| 精确 ID | "ID xxx 的卡片" | `knowledgeCard.findUnique` |
| 关键词 | "关于宿舍分配的卡片" | `knowledgeCard.findMany` 按 `summary` / `body` 搜索 |
| 批量描述 | "所有 domainTag=新生入学 且提到 2024 的卡" | 组合条件查询 |

### Step 2: 查询当前卡片

至少查这些字段：

- `id`
- `summary`
- `body`
- `sourceExcerpt`
- `sourceUrl`
- `sourceDescription`
- `sourceType`
- `verificationStatus`
- `domainTag`

若找到多张候选卡，先向用户列清单，再继续。

### Step 3: 根据错误类型确定流程

#### 3a. 拼写/断句/格式错误

可直接修复，无需外部验证。改完回查即可。

#### 3b. 事实性错误（与 sourceExcerpt 矛盾）

1. 对照 sourceExcerpt 确认差异
2. 以 sourceExcerpt 为准修正
3. 若 sourceExcerpt 本身有误→报告用户，请求授权联网核对

#### 3c. 来源信息遗漏

1. 逐句比对 sourceExcerpt 和当前 body
2. 将遗漏的关键事实补入对应卡片（适用横切知识点规则）
3. 做逐句覆盖校验确认无更多遗漏

#### 3d. 时效性过时

Agent 无法单方面判断一条信息是否过期。以下情况才触发"过时"修正：
- 用户在请求中明确说明了新政策的生效时间或内容
- sourceExcerpt 中已有新旧对比信息
- 用户授权联网核对且找到可靠的官方更新

不准在没有外部依据的情况下，自行将某条信息标记为"过时"并修改。

### Step 4: 若问题涉及图片，先做图片归属审计

#### 4.1 抽取当前图片

从 `sourceExcerpt` 中提取所有本地图片路径：

```text
/knowledge-images/xxx.png
```

#### 4.2 查复用情况

对每张图片，查询还有哪些卡引用了它。

重点看两件事：

1. **是否被其他 `sourceUrl` 的卡引用**
2. **是否明显属于别的主题组**

若答案是"是"，优先判定为**高风险错图**，不要直接沿用。

#### 4.3 查同来源候选图

查询同一个 `sourceUrl` 下的其他卡，找出：

- 该来源已经使用过的图片
- 同一批制卡产物中的相关截图
- 与当前卡语义最接近的局部图

优先选择：

- 培养方案结构图
- 课程类别图
- 表格、流程图、操作截图

排除：

- logo
- 头像
- 装饰图
- 与当前卡主题无关的截图
- 仅与其他卡主题强相关的截图

#### 4.4 必要时回看批量制卡痕迹

如果仓库里有批量制卡流程，可借用：

- `docs/CARD-WORKFLOW.md`
- `agent-workflows/card-batch/runs/`
- 与当前来源对应的 `cards.json` / 审稿记录 / iteration 记录

重点判断问题出在：

- 原始制卡阶段
- 后续补图/修图阶段
- 人工复用旧图阶段

#### 4.5 若必须访问原网页核对

只有在本地证据不足、需要精确确认"原网页到底是哪几张图"时，才请求用户授权联网核对。

### Step 5: 向用户说明修改方案

至少明确：

- 要改哪张卡
- 改哪些字段
- 旧图是什么
- 新图是什么
- 为什么判断旧图错了

图片修复方案建议写成：

```text
修改方案：
- 移除当前 sourceExcerpt 末尾的 7 张错图（来自其他 sourceUrl 的 jwinfo 主题图）
- 替换为同一 sourceUrl 下、与"课程类型/培养方案结构"直接相关的 3 张图
- 其他字段保持不变
```

### Step 6: 执行修改

用 Prisma 直接更新目标字段。常见字段包括：

- `summary`
- `body`
- `sourceExcerpt`
- `sourceUrl`
- `sourceDescription`
- `sourceType`
- `verificationStatus`
- `domainTag`
- `archivedAt`

执行长中文正文、长 `sourceExcerpt` 或多字段更新时：

1. **不要用 `node -e "const body='长中文...'"` 直接承载正文**。在 Windows/PowerShell 下容易出现中文乱码、命令长度截断、引号被剥离等问题。
2. 优先用 `update-card-fields.js` 从环境变量或 JSON 文件读取待更新字段；修改前可先不加 `--apply` 看 dry-run。
3. 如果必须临时写 Prisma 脚本，长文本也要通过环境变量或 UTF-8 JSON 文件传入，避免把正文嵌入命令行字符串。
4. 写入失败时先回查数据库当前值，确认问题后换一种写库方式重试。连续 2 次失败则向用户报告错误并停止。

### Step 7: 修改后立即验证

至少验证以下几点：

1. 数据库中的目标卡已更新
2. 目标卡不再引用错误图片
3. 新图片与当前 `sourceUrl` 一致，且语义上服务于当前卡
4. 若是批量修正，输出成功/失败统计
5. 若更新了中文 `summary` / `body`，额外验证标题不包含异常 `?`，正文包含预期关键词，且来源字段未被误改。
6. 若是拆分操作，验证所有新建卡片均已写入、来源字段一致、老卡已正确归档。

## 图片修复专门规则

1. **先查 `sourceUrl`，再看图片**。不要只看图片本身像不像。
2. **跨来源复用要默认高风险**。同一张图若出现在多个 `sourceUrl` 下，先假设其中至少一个地方用错了。
3. **同来源复用可以接受，但要看语义是否匹配**。同源不等于同题。
4. **优先保守修复**。本地无法 100% 还原原网页时，宁可只保留最确定的 2-4 张相关图，也不要把可疑图继续塞进去。
5. **不要为了"有图"而硬配图**。没有可靠图就留空，必要时标 `NEEDS_REVIEW`。
6. **图里的事实若要参与回答，必须另写进 `body`**；`sourceExcerpt` 中的图只负责溯源展示。

## 常见修正场景

### 内容过时

只有以下情况触发：
- 用户明确给出新政策的生效时间或内容
- sourceExcerpt 中含新旧对比
- 用户授权联网后找到可靠官方更新

流程：
1. 定位相关卡片
2. 更新 `body` 中的时效信息，在变更处标注当时的依据
3. 标 `verificationStatus: "NEEDS_REVIEW"`

### 事实性错误

1. 对照 sourceExcerpt 确认差异
2. 以 sourceExcerpt 为准修正
3. 若 sourceExcerpt 本身有误，报告用户

### 来源信息遗漏

1. 逐句比对 sourceExcerpt 和 body
2. 补入遗漏事实，遵守横切知识点规则
3. 做逐句覆盖校验

### 图片错配

1. 审计当前图片引用
2. 查跨来源复用
3. 从同来源候选图中重选
4. 更新 `sourceExcerpt`
5. 回查验证

### 需要归档

1. 定位卡片
2. 设置 `archivedAt: new Date()`
3. 确认归档后不再参与正常检索

### 合并重复

1. 查出重复卡
2. 选保留卡
3. 合并独有信息
4. 归档其余卡片
5. 确认无信息丢失

## 注意事项

- `sourceExcerpt` 是逐字原文和图片溯源材料，修改时不要夹带新的 AI 解释。
- 图片错配问题往往不是前端显示问题，而是 `sourceExcerpt` 中的本地图片引用写错了。
- 批量修正前，至少保留目标卡 ID 列表和原始查询条件。
- 写库失败时先回查再重试，连续 2 次失败则停止并报告。
- 凡涉及拆分/合并操作，修改后必须做完整性校验（逐句覆盖或信息丢失检查）。
