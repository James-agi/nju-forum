# 语义检索——最小验证方案

> 这份文档只回答一件事：**“按意思找卡”到底能不能把现在漏掉的卡捞回来。**
> 在花力气改生产代码之前，先用一个临时脚本证明方向对，再谈正式接入。

---

## 0. 背景（为什么做这个）

现在的检索是**按字面找**：卡里没出现问题里的词，就搜不到。
实测翻车案例：

- 问“转专业” → 漏掉「转CS的官方流程」「转数学有几种途径」「计算机科学与技术转入条件」——这些卡用的是“转CS/转入/转数学”，标题正文都没有“转专业”三个字。
- 问“转计算机有没有建议” → 证据评估判 `UNRELATED`，答不上。

“按意思找”（语义检索 / embedding）能不靠手工同义词，直接把意思相近的卡拉到一起。
**本方案验证它在这 320 张卡上是否真的有效，不做任何生产改动。**

---

## 1. 验证目标（可证伪）

跑完脚本后，对照下面这张表，**有就是过，没有就是不过**：

| 问题 | 关键词检索现状 | 语义检索期望 |
|---|---|---|
| 转专业 | 漏掉转CS/转数学/CS转入条件 | top5 里至少捞回 1 张这类卡 |
| 转计算机有没有建议 | 证据不足，答不上 | top5 里出现转CS流程/CS转入条件 |
| 大类怎么分流 | （执行时实测） | top5 与问题明显相关 |
| 一卡通丢了怎么办 | （执行时实测，精确名词，看会不会反而变差） | 仍能命中一卡通相关卡 |

最后一行是**反向检查**：语义检索对“一卡通”这种精确名词，不能比关键词更差。如果变差，说明必须“关键词+语义”两条腿走，不能纯语义。

---

## 2. 阶段一：最小验证（不碰数据库、不碰现有代码）

### 2.1 前置硬门槛：先确认 embedding 接口可用

这是**第一步，也是能不能往下做的开关**。

现有 `llm-config.ts` 配的是 chat 模型（deepseek-v4-flash 之类），**chat 模型不等于 embedding 模型**。必须确认代理后端提供了一个可用的 embedding 模型。

做法：写一个最小探针，对 `${baseUrl}/embeddings` 发一次请求，body 形如：

```json
{ "model": "<待确认的embedding模型名>", "input": "测试一句话" }
```

- 能返回一个数字数组（向量）→ 通过，记下模型名和维度（长度），继续。
- 报 404 / 模型不存在 / 不支持 → **停**。此路代理不提供 embedding，需要先解决“去哪算向量”（换一个支持 embedding 的 key，或本地模型）。不要硬往下做。

常见 embedding 模型名供试探：`text-embedding-3-small`、`bge-m3`、`text-embedding-v3`。具体哪个可用，由探针结果决定，**不要假设**。

模型名建议读新环境变量 `KNOWLEDGE_EMBEDDING_MODEL`，复用现有 `KNOWLEDGE_LLM_API_KEY` / `KNOWLEDGE_LLM_BASE_URL`。

### 2.2 把 320 张卡变成向量，存成一个文件

新建临时脚本 `temp-embed-cards.ts`（`temp-*` 已 gitignore，验证完即删）：

1. 用 Prisma 读出所有 `archivedAt: null` 的卡，字段取 `id / summary / body / domainTag / verificationStatus`。
2. 每张卡拼一段待编码文本：`summary + "\n" + body`（domainTag 可加在前面）。
3. 逐张调 `/embeddings` 拿向量。注意：
   - **限速**：串行或小并发（≤3），别一次性轰 320 个请求把代理打挂。
   - **省钱省时**：把结果写进 `temp-card-vectors.json`（结构 `[{id, summary, vector}]`）。脚本要能**断点续跑**——已经在 json 里的 id 跳过。（这条是你自己的“随转随存”原则。）
4. 跑完确认 json 里有 320 条向量。

### 2.3 拿翻车问题重搜一遍

新建 `temp-semantic-search.ts`：

1. 读入 `temp-card-vectors.json`。
2. 对每个测试问题（见第 1 节表格）：
   - 调 `/embeddings` 把**问题**也变成一个向量。
   - 和 320 张卡的向量逐一算**余弦相似度**，从高到低排序。
   - 打印 top5：相似度分数 + summary。
3. **并排对照**：同一个问题，再调一次现有的 `retrieveKnowledgeCards()`（关键词检索），把两边 top5 打到一起，方便肉眼对比谁捞回了漏卡。

余弦相似度就是：两个向量点积除以各自模长。一个十行的小函数，不需要任何库。

### 2.4 验收（你自己看）

对着第 1 节的表逐行看 top5：

- ✅ 语义检索把关键词漏掉的卡捞回来了，且“一卡通”这种精确词没变差 → **方向成立，进阶段二**。
- ⚠️ 语义捞回了漏卡，但精确词变差了 → **方向成立，但必须两条腿**（关键词+语义一起），记下来。
- ❌ 语义也没捞回漏卡，或捞回一堆不相干的 → **方向不成立**，问题在别处（可能是卡本身没写、或测试问题太刁），别正式接入，回来重新诊断。

## 3. 阶段二：验证通过后才做（先给方向，细节等结果）

> 阶段一不通过，**不要进这一步**。

1. **数据库装向量能力**：PostgreSQL 用 pgvector 扩展，给 `KnowledgeCard` 加一列 `embedding`。
   - 需先确认你的 PG 是否允许 `CREATE EXTENSION vector`（托管数据库可能不让，届时再议存法）。
2. **建卡/改卡时顺手算向量**：卡内容变了就重算这一列。
3. **检索改成两条腿**：关键词召回一批 + 语义召回一批，合并去重，再按分数排。**不要直接删掉关键词检索**——精确名词还得靠它。
4. **红线不动**：答案仍然必须带引用、不许编、不混源。语义检索只是“多捞几张候选卡”，最终答案照旧只能来自卡。

---

## 4. 明确不做的事

- 不在验证阶段装 pgvector、不改 schema、不改 `ask/route.ts`、不改 `retrieval.ts`。
- 不删除现有关键词检索。
- 不动反问层 / 证据门槛——那是另一条线的病（反问抢跑、门槛偏严），和检索方式无关，本方案不碰。
- 不把向量入生产库，验证阶段只用 `temp-card-vectors.json`。

---

## 5. 风险与兜底

| 风险 | 兜底 |
|---|---|
| 代理不提供 embedding 接口 | 阶段一 2.1 直接拦下，不浪费后续功夫 |
| 语义召回噪音（捞回“听着像其实不沾边”的卡） | 靠现有红线兜：必须引用、不许编；阶段二保留关键词做精确锚点 |
| embedding 一次性请求过多被限流 | 串行/小并发 + 断点续跑 json |
| 换了语义检索，反问抢跑/门槛过严照旧 | 已声明本方案不解决这两个病，别指望换检索一了百了 |

---

## 6. 一句话总结给执行 agent

先用探针确认 embedding 能调（不能就停）；能调就把 320 张卡和几个翻车问题都变成向量、算余弦相似度、并排对比关键词检索；**只动 `temp-*` 脚本，碰一行生产代码都算越界**。结果交给作者本人拍板。
