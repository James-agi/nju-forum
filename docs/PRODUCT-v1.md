# NJU 知识库 · 产品文档（v1.0）

> 这份文档回答三件事：**本心是什么**、**P0 做什么**、**什么坚决不做**。
> 它取代 v0.3。v0.3 把产品定位成"一体化学习生产力系统"，做成了 4 个平行模块，是定位错误，已废弃。
> 本文档的每一条判断，都来自一次被真实痛点例子逼出来的对话，不是凭空设计。

---

## 0. 为什么推翻 v0.3（先认错，再往前）

v0.3 的定位是"把讨论/读书/作业/计划四件事收拢到一个工作台"。它有四个硬伤，必须记住，否则会重蹈覆辙：

1. **定位偏离本心**：作者的本心是"解决日常交流的信息无法沉淀"，不是"做一个学习生产力工具"。
2. **P0 严重过载**：4 个模块每个都是毕设级工作量，单人做不完。
3. **模块零凝聚力**：本质是"4 个独立 app 摆在同一个 URL"，不是一个产品。
4. **痛点全靠想象**：v0.3 的痛点描述都"看起来合理"，但没有作者自己的真实数据支撑。

v1.0 用一句话纠正：**论坛/信息沉淀是骨架，其它能力（如果有）是这根骨架上长出来的，不是平行摆放的。**

---

## 1. 本心（产品的唯一轴心）

> **NJU 里有大量只有"经历过的人"才知道的经验和事务知识——怎么入学报到、三三制怎么修、保研要满足什么、某个比赛怎么入门。这些信息存在于人的脑子里、散在各个 QQ 群和零散文档里，没有一个地方把它们收集起来、让后来人能主动检索到。**
>
> **产品要做的，是把这些信息从"散落、易丢、要重复问人"的状态，变成"留得下、找得到、可溯源"的状态。**

三个动词定义了"成功沉淀"：

| 动词 | 含义 | 反面（现状） |
|---|---|---|
| **留下来** | 信息被结构化存进知识库，不随聊天流水冲走 | 通知刷过去就没了，经验说完就忘了 |
| **被找到** | 新人能主动检索，不必再开口问人 | 多群辗转搜，搜不到只能问学长 |
| **被引用** | 答案能溯源到具体出处，知道这话从哪来 | 口耳相传，没人知道哪句可信 |

**本心的边界（防止再次跑偏）**：
任何功能，只有当它服务于"信息留下来/被找到/被引用"时，才属于这个产品。一个功能如果只是"对学生有用的学习工具"，但和信息沉淀无关，那它是**另一个产品**，不进这里。教材问答、作业看板、ToDo —— 都是在这条边界外被砍掉的。

## 2. 真实痛点（作者自己的例子，不是想象）

v1.0 的定位是被下面这些**真实发生过的事**逼出来的，不是设计出来的。记录在此，作为一切判断的锚点。

| # | 真实事件 | 信息类型 | 怎么丢的 / 卡在哪 |
|---|---|---|---|
| 1 | 想知道校园跑截止时间，在多个 QQ 群搜聊天记录辗转才找到 | 事务通知 | 通知刷过去，自己记录里找不到，搜不到只能问人 |
| 2 | 想入门 RoboMaster，问学长问多了不好意思，用 AI 也看不懂 | 学长经验 | 不好意思继续问，经验没流通 |
| 3 | 转 AI 方向前找面试经验，学长说"没有，因为不在意" | 学长经验 | 经验存在但从没被沉淀，问也问不到 |
| 4 | 选导师进科研，到处翻老师科研经历，看不懂，没人帮筛选 | 公开信息+经验 | 信息散、没有人帮解读 |
| 5 | 看到音乐节活动后来忘了，从群聊翻起，有时直接错过 | 事务通知 | 通知型信息没有提醒机制 |

**作者自评"最难受"的是学长经验类**，理由原话：「这个我自己是真不知道，也没有其他渠道去找」。进一步追问，最痛的那一层是 **丙：没有一个地方能让我主动去找、不用开口问人**。

**一个必须诚实记下的张力**：本心一开始被表述成"信息沉淀"（像是一个容器问题），但真实痛点里有相当一部分是"经验根本没被说出来"（一个内容生产问题）。v1.0 的应对是：**P0 先不依赖任何人贡献，由作者自己把已存在的信息填进去**（见 §4、§6），把"让别人来贡献"推迟到内容已经够厚之后。

## 3. 它到底是什么（一句话 + 反面定义）

**它是：一个 NJU 知识库 + 一个会溯源的问答 Agent。**

- 用户问一个具体问题 → Agent 在知识库里检索 → 给出**带出处、可跳转**的回答；
- 检索不到 → **明说"暂时没人沉淀过这个问题"**，并把这个问题记入"缺口库"；
- 知识库的内容**先由作者一个人填**，不靠任何人自愿贡献来启动。

**它不是**（反面定义同样重要）：
- 不是聊天陪伴机器人（不需要记住"你是谁"，见 §8 对 Memory 的澄清）；
- 不是套壳 ChatGPT（核心区别就是**溯源 + 不编造**，见 §5）；
- 不是又一个信息聚合站（聚合没有护城河，见 §7）；
- 不是论坛（论坛是 P2 才开放的 UGC 形态，不是 P0）。

## 4. 数据地基：知识卡片

这是整个产品的核心数据单元。**所有功能都围绕它转。**

### 4.1 为什么是"卡片"不是"文章"

知识按**小颗粒**存储（一条卡片 = 一个独立知识点），不按整篇文章存。原因直接来自本心红线"精确溯源"：

- 整篇存 → 溯源只能指到"这篇文章"，指不到具体哪句，答案糊；
- 切成卡片存 → 溯源精确到点，检索命中率高，答案准。

代价是录入更费劲（要把长文档拆成知识点），但这个代价靠"AI 帮你拆、你审核"来缓解，且正好落在红线"用户最终决定权"上。

### 4.2 卡片的三层展示

一张卡片有三层深度，对应三种用户动作：

```
第一层  摘要      → 列表里默认看到的一句话，扫一眼就知道是不是要找的
第二层  正文      → 点击展开，看到这条知识的完整内容
第三层  原始出处   → 跳转按钮，去到搜集来源的原文（官网/文档链接）
```

### 4.3 字段草案（P0 够用，故意不过度设计）

```
KnowledgeCard
  id
  摘要              一句话概括，列表展示用
  正文              展开后看到的完整知识（小颗粒，对应 §4.1）
  来源原文链接       跳转按钮指向的 URL；可空（有些来源是"问学长"，没链接）
  来源说明           文字描述来源：官网 / 某文档 / "问了XX学长"
  领域标签           如"新生入学"；不限枚举，开放（见 §6 的"能力不设限"）
  是否已核实         作者录入时标注，承接"内容可信度"问题
  创建时间 / 更新时间
  (预留) 关联卡片     给以后知识点之间连关系留位置，P0 不实现
```

设计说明：
- "来源"故意拆成**链接**和**文字说明**两个字段。来源有链接时跳转按钮可用；来源是"问学长"无链接时按钮自然不显示，不会出现"点了跳转是空"的尴尬。
- "是否已核实"承接作者关心的**内容可信度**问题。P0 内容全由作者亲手录、亲手审，可信度由作者本人背书；这个字段让"未核实"的卡片可被标记区分。
- "关联卡片"只预留位置不实现，避免 P0 过度设计。

## 5. 核心红线：溯源与不编造（产品的技术尊严）

这是 v1.0 区别于"随便套个 ChatGPT 网页"的**唯一**技术尊严，也是生死线。

**为什么是生死线**：用户来问"保研要满足什么""第二专业怎么修"这种问题。如果 AI 编一个看起来合理但错误的答案，用户照做，耽误的是人家的保研、转专业——这种错误有现实后果，用户被坑一次就永不再信。

### 5.1 技术路径：RAG（检索增强生成）

```
用户问题
  → 在知识库里检索最相关的若干张卡片（retrieval）
  → 把这些卡片喂给 LLM，约束它"只能基于这些卡片回答"（augmented）
  → LLM 生成答案，并标注每个结论来自哪张卡片（generation + citation）
  → 检索结果为空 / 不相关 → 走 §5.3 的"没命中"流程
```

### 5.2 三条铁律

1. **精确溯源**：答案的每个结论都要能指回知识库里的某张卡片。指不回去的内容，不输出。
2. **无依据拒答**：检索不到足够依据时，明确说"知识库里没有相关内容"，不允许补全式编造。
3. **不允许通用知识兜底**：知识库没命中时，**禁止**用 LLM 自身训练知识来回答，哪怕标注"非本库内容"也不行。
   - 这一条是作者明确拍板的。它定义了产品的"人格"：宁可显得笨，也不说没出处的话。
   - 注意：这与 v0.3 里"精华问答 agent 无命中时可用一般知识"的旧设定**相反**。v1.0 取消那个例外。

### 5.3 没命中时怎么办（缺口闭环）

没命中**不是死路**，而是产品的增长燃料。完整闭环：

```
用户问问题
  ├─ 知识库命中 → 溯源回答 ✓
  └─ 没命中
       → 回复用户：「暂时没人沉淀过这个问题，已经记下了」
       → 把问题存进"缺口库"
       → 作者（P0 阶段的人肉填充者）针对缺口补一张卡片
       → 系统主动推送给当初提问的人：「你之前问的 X，现在有答案了」
```

这个闭环把"答不出"从尴尬变成"挂个单，有货了叫你"。它解决了 P0 知识库薄、很多问题答不出的体验问题。

**闭环的两个依赖（必须清醒）**：
- 推送要成立，需要"用户身份"——现有 NextAuth 登录已提供基础，不是新坑。
- "补答案"这一步在 P0 **由作者本人完成**（人肉客服）。这是唯一能让闭环在单人、零其他用户的情况下真的转起来的方式。指望真实学长自愿来补，是 P1 以后的事（见 §7、§9）。

## 6. P0 范围（最后一道防止过载的闸）

### 6.1 P0 功能清单

```
P0 必须有：
  1. 知识卡片的录入        作者自己录，含来源标注、是否核实
  2. 问答 Agent           检索 → 溯源回答 → 没命中则记入缺口库
  3. 缺口库               收集答不出的问题，供作者看"该补什么"
  4. 用户登录             复用现有 NextAuth

P0 不做（挂起，见 §7 路线图）：
  论坛 / 自愿贡献              → P2
  推送"你问的有答案了"          → P1（依赖闭环先被作者手动跑通）
  时效性数据（DDL / 活动）       → P1+
  用户浏览 / Wiki 模式          → P1
  外部内容安全审查              → P2（P0 只有作者一人录，无此风险）
  Agent 个性化记忆（Memory）     → P1（见 §8 澄清，P0 不需要）
  教材问答 / 作业看板 / ToDo     → 砍（属于另一个产品，违反 §1 本心边界）
```

### 6.2 冷启动的内容策略：能力不设限，精力有侧重

这是作者反复想"全部录入、覆盖多方面"与"单人半个月做不完"之间的张力，最终的拍板：

- **数据结构不做领域限制**：卡片的"领域标签"字段开放，录什么方面都能装。满足"用户问啥都行、搜到啥都能录"。
- **但 P0 精力集中在一个方面**：**新生入学事务**。
  - 选它的理由：全员刚需、问题高度重合（一次录入覆盖一整届）、复用率最高、且大多是稳定的制度/流程知识，作者一个人就能录。
  - **只录稳定部分**（三三制、选课规则、报到流程、院系结构等），**强时效部分**（今年具体报到时间、今年迎新安排）挂 P1。

> 算账逻辑（记住，别再想"全部录入"）：半个月总量固定。摊到 8 个方面 = 处处稀薄、处处失望；集中到 1 个方面 = 这一块真能打、真留得住人。"某方面真好用"能留人，"啥都浅尝辄止"留不住人——这正是 v0.3 翻车的同一个病根。

### 6.3 时间盒

**P0 第一个能用的版本：约半个月。** 这个时间盒反过来检验 P0 清单——如果半个月做不完，砍的是功能数量，不是质量。

## 7. 护城河与路线图（清醒地面对"凭什么是你"）

### 7.1 护城河的真相

作者选定的护城河是 **C：网络效应**（用的人越多，缺口暴露越多，内容越厚，后来者越难追）。

**但必须清醒**：网络效应是**终点，不是起点**。单人开发者一开始没有网络效应，强行追求它就会掉进冷启动死亡循环：

```
没有内容 → 用户来了没东西看 → 用户走 → 没人贡献 → 还是没有内容
```

**破局方式**：不靠"建好场子等人来"，而是**作者自己先把内容填满**（§5.3、§6.1 的人肉填充）。先做到"哪怕只有作者一个人用也成立"，再谈滚雪球。这正是产品红线"单用户先成立"。

已知的薄弱点（诚实记录，不粉饰）：
- 信息聚合 + AI 问答这个模式本身**没有护城河**，谁拿公开数据 + LLM 都能复制。
- 真正的壁垒只可能来自"作者持续填充 + 缺口数据驱动 + 未来 UGC 沉淀"形成的**内容资产**，不来自技术。

### 7.2 路线图

| 阶段 | 内容 | 验收信号（真实使用，不是功能数） |
|---|---|---|
| **P0** | 知识卡片录入 + 溯源问答 + 缺口库 + 登录；内容集中在"新生入学稳定知识" | **作者自己每天离不开它**，遇到 NJU 信息问题宁可问它，不再多群辗转搜 |
| **P1** | 缺口推送闭环自动化 + 用户浏览/Wiki 模式 + Agent 个性化（推送给提问者）+ 时效性数据试点 | 至少一个同学愿意稳定使用并给反馈 |
| **P2** | 开放他人贡献（论坛/UGC 形态复用现有论坛代码）+ 外部内容安全审查 + 网络效应真正启动 | 有他人主动贡献的卡片被检索/引用 |

**成功的定义（写给一年后的自己）**：
- ❌ 错误标准："很多 NJU 同学在用"——单人项目用这个标准必然自我怀疑然后放弃。
- ✅ 正确标准（P0）："**作者自己每天在用它，它真的省了多群搜、不好意思问学长的事。**"
- 如果连作者自己都宁可继续翻 QQ 群——产品是死的，应该停。如果作者自己离不开了——已经成功一半，剩下的是扩散问题。

## 8. 关键澄清：知识库数据结构 ≠ 记忆系统

作者提过"需要规划记忆系统架构"。这里纠正一个概念混淆，因为它直接影响 P0 不被挂上重物：

| 概念 | 是什么 | P0 需要吗 |
|---|---|---|
| **知识库数据结构** | 卡片怎么存：字段、颗粒度、卡片间关联（即 §4） | **必须有，这就是 P0 的地基** |
| **记忆系统（Memory）** | Agent 记住"和某用户聊过什么、用户偏好、多轮上下文" | **P0 不需要** |

P0 的产品是"问一个问题 → 检索知识库 → 溯源回答"，它**不需要记住用户是谁**。Agent 个性化记忆挂到 P1（等要做个性化推送时再说）。P0 真正要规划的，只有 §4 的卡片数据结构。

## 9. 待决问题（不假装都想清楚了）

**内容侧**
- 作者人肉填充的可持续性：半个月后能否保持每周补卡片的节奏？这是 P0 成败的真正变量，比任何技术都关键。
- 缺口库到"补卡片"之间，P1 怎么从"作者补"过渡到"学长补"？过渡机制未定。
- 来源是"问学长"的卡片，可信度怎么对用户表达？目前靠"是否已核实"字段，是否够？

**技术侧（属工程文档，此处仅挂标记）**
- 检索用关键词还是向量（语义）检索？现有库是 PostgreSQL，可上 pgvector 做语义检索，无需换库。具体在工程文档定。
- 知识库数据库的完整 schema 设计（§4 只是字段草案，不是最终 schema）。
- 现有论坛 6 张表（User/Post/Reply/Section/Tag/Favorite）在新定位下的处置：P0 不动它，作为 P2 开放 UGC 时的地基。

**外部依赖**
- LLM 调用成本：每月上限、谁付、超额怎么办。沿用"单一 OpenAI 兼容接口，baseURL+apiKey 从 env 读，换模型只改配置"。

## 10. 非目标（明确不做，提议时先看这一节）

- **不做平台爬取**：作业、通知、成绩等全部靠作者录入或 AI 从用户上传载体抽取。不做模拟登录抓教务。
- **不做通用知识兜底**：知识库没命中就拒答，不用 LLM 自身知识糊弄（§5.2 第 3 条）。
- **不做混源问答**：不同来源的卡片不在同一回答里混合引用，每个回答的依据可解释。
- **不做开放域闲聊**：会稀释"NJU 信息沉淀"这个核心。
- **不做教材问答 / 作业看板 / ToDo**：这是 v0.3 的平行模块病，属于另一个产品。
- **不做记忆型聊天陪伴**：P0 不记住用户（§8）。
- **不做付费 / 会员 / 商业化**。
- **不在 P0 做论坛 / UGC**：内容先靠作者填，开放贡献是 P2。
