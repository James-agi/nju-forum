# 卡片工作流——使用与迭代

> 这份文档回答的不是"一张卡怎么做"（那是 [CARD-WORKFLOW.md](./CARD-WORKFLOW.md) 的事），
> 而是**"你和 AI 怎么配合，把做卡这件事变成一条越用越顺的流水线"**。
> 一句话定位：CARD-WORKFLOW 是"施工规范"，这份是"怎么指挥施工、怎么让施工队越来越懂你"。

---

## 1. 这套工作流到底在解决什么

最初的想法是"让 AI 自动抓取网页、总结成卡、我审一下就自动入库"。这个想法的**致命陷阱**是"审一下就自动入库"——批量自动跑的时候，人会从"审查"滑成"橡皮图章"，给一堆没真看过的卡盖"已核实"章。那正好踩中产品最该死的错（见 CARD-WORKFLOW §2）。

所以这套工作流的核心不是"自动化"，是**分工**：

| 谁 | 干什么 | 为什么是他 |
|---|---|---|
| **AI** | 读源、提取事实、结构化、对齐语气、守机械红线、写进库（标 NEEDS_REVIEW） | 这些是重复劳动，机器快且不累 |
| **你** | 判断哪张卡值得做、审草稿、贡献过来人洞察、把 NEEDS_REVIEW 升 VERIFIED | 这些是判断，长在你的经历和品味里，AI 给不了 |

**边界**：AI 干 80%（感知 + 结构 + 语气），你守那 20%（判断 + 洞察 + 拍板）。
那 20% 恰恰是产品凭什么比直接读语雀强的唯一理由。**别追求 AI 干到 100%——真到那天，说明产品已经退化成原文的复述。**

## 2. 当前闭环：五步（路线 A，零新代码）

```
① 你扔一个源        ——  一张图 / 一段文字 / 一个链接
② AI 出三层卡草稿    ——  对齐录入器字段，verificationStatus 一律先 NEEDS_REVIEW
③ AI 写进库          ——  用 Prisma 直接写（接口有登录态鉴权，AI 拿不到你的 session）
④ 你审              ——  在 /admin/knowledge 列表点开这张 NEEDS_REVIEW 卡
⑤ 通过 / 退回        ——  通过→你把状态升 VERIFIED；退回→把理由记进 CARD-WORKFLOW §5
```

**为什么是"路线 A"**：这条路不需要任何新代码，今天就能跑。AI 把卡写成 `NEEDS_REVIEW` 状态进库，你在已有的管理页审——录入器、列表、状态筛选都是现成的。

**③ 为什么 AI 用 Prisma 写而不是调 HTTP 接口**：`POST /api/knowledge/cards` 有 `requireKnowledgeAuthor()` 鉴权，需要登录 session，AI 这边拿不到你的 cookie。所以 AI 走 Prisma 直接写库，挂在你的 admin 账号下（`createdById`）。效果对你完全一样：库里多一张待审的卡。

**③ 的硬约束**：AI 写进去的每一张，`verificationStatus` 必须是 `NEEDS_REVIEW`。这是铁律——AI 没有权力给卡盖"已核实"章，那是 ④⑤ 你的事。

## 3. 怎么审一张草稿（你的检查清单）

按这个顺序过，任何一项不过就退回：

**第一关 · 红线（违反直接退，不用往下看）**
- [ ] 有没有**编造**？凡是来源里没有的事实（学分、时长、地址、时间），AI 是不是瞎补了？——对照 sourceExcerpt 查。
- [ ] 有没有**混旧版**？只该录当前级，旧版本信息混进来没有？
- [ ] **年份**标了没？sourceDescription 里有没有版本/年级标注？body 结尾有没有**误加**"（信息适用于X级方案）"这类标注？（body 结尾不加年份标注，见 CARD-WORKFLOW §2 铁律3）

**第二关 · 硬指标（拦平庸）**
- [ ] body 里有没有**至少一句原文没有的话**（经验排序/踩坑/时机/澄清）？没有就退，哪怕字字属实。——这条是 CARD-WORKFLOW §2.5。

**第三关 · 语气（拦爹味）**
- [ ] 是不是**过来人平视**？有没有滑回"务必/切记/千万"的祈使句、套【】标签？
- [ ] 一句话检验：**像学长私下说的话，还是像辅导员发的通知**？后者退回。

**第四关 · 字段（拦填错）**
- [ ] domainTag 对不对？（现有卡都挂"选课规则"这类**细标签**，别用"新生入学"这种泛标签）
- [ ] sourceType 对不对？（官方文件 OFFICIAL / 整理文档 DOCUMENT / 问学长 SENIOR / 作者经验 AUTHOR_EXPERIENCE）
- [ ] sourceExcerpt 是不是逐字原文、没夹带 AI 的话？
- [ ] 如果原文信息在图片里，sourceExcerpt 有没有放入下载后的本地图片引用？图里的事实有没有另写进 body？

**第五关 · 查重复核（防悄悄造重复卡）**
- [ ] 起草侧那句"现状：库里已有X/Y ／ 无相关卡"的声明，**交了没**？
  - 没交 = 它根本没查库 → 打回，让它先查（见 [CARD-WORKFLOW.md §1 第0步](./CARD-WORKFLOW.md)）。
  - 交了 → 核一下：这张卡跟它列出的已有卡，是不是其实在讲同一件事？是 → 该改成补全那张旧卡，而不是新建。
- [ ] 这一关你是**复核**它查过没，不是替它查。执行者不具备查库能力时（首次交接新模型常见），这一关就由你亲自查库兜底。

全过 → 把状态从 NEEDS_REVIEW 升到 VERIFIED，这张卡正式生效。

## 4. 退回理由怎么记

退回不是终点，是**喂给流水线的饲料**。每次退回，把理由追加到 [CARD-WORKFLOW.md §5 退回案例库](./CARD-WORKFLOW.md)，用那边约定的格式：

```
### YYYY-MM-DD ｜ 卡片：<简述>
- 源：<什么来源>
- 初稿问题：<哪里不行——违反铁律？平庸？爹味？字段错？>
- 修正方向：<改后该长啥样>
- 提炼规则（可选）：<这次教训能不能升成通用规则>
```

**记的时候多花十秒想一层**：这次的问题是"这一张卡特有的"，还是"AI 会反复犯的"？
- 一次性的（比如这张卡某个数字看错了）——记一笔就行。
- 反复的（比如 AI 总爱把"建议"写成"务必"）——这就是该升进正文规则的信号，见 §5。

## 5. 怎么迭代（让流程越用越准的唯一机制）

先认清一个事实：**这套流程里能"变强"的，只有文档，不是 AI。**

- **AI（模型）**：不变。你退回多少次，它下次还是同一个它，不会"记住"上次的教训——除非教训写进了它每次都会读的文档。
- **你的品味**：会越来越利，但它在你脑子里，不在流程里，没法直接复用。
- **CARD-WORKFLOW 文档**：**唯一能跨会话沉淀的东西**。AI 每次起草前读它，文档里多一条规则，AI 的产出就多守一条线。

所以迭代机制很简单：

```
退回（§4 记理由） → 攒一批 → 回看哪些原因反复出现 → 提炼成规则升进 CARD-WORKFLOW 正文 → AI 下次起草自动遵守
```

**AI 主动迭代**：AI 每次录入完成后，自动执行迭代动作（记录退回案例、提炼规则、检查工作流），不需要用户提醒。详见 `AGENT-CARD-PROMPT.md §Step 6`。

**节奏建议**：每跑 5-10 张卡，回看一次退回案例库。把出现 ≥2 次的同类问题，写成一条正文规则。不用频繁做，攒够了再提炼，否则规则会碎。

**工作流检查**：每次录入后，AI 自动检查文档规则是否一致、工具是否正常、是否有过时描述。发现问题立即修复并记录。这是防止规则漂移、保持流程健康的关键动作。

**警惕一个失灵信号**：流程越跑越顺之后，你的退回会越来越少。这未必是好事——可能是 AI 学会了"挑不出毛病"，但"挑不出毛病"不等于"好"。那种"没错但没灵魂"的平庸卡最容易蒙混过关。这就是为什么 §3 第二关那条"至少一句原文没有的话"是**正向**指标：它逼每张卡都得贡献洞察，而不只是"不犯错"。审到后期，重点从"挑错"转向"问这张卡值不值得存在"。

## 6. 什么时候从路线 A 升级到路线 B

**路线 B** = 在 admin 里做个"贴图/贴链接 → AI 自动填充草稿表单 → 待审"的专用入口。它要写新代码、可能要接多模态模型 API、要处理图片上传。

> 2026-06-01 补充：现在已经做了一个更小的“图片溯源工具”——后台卡片编辑器和 `fetch-yuque.ts` 能从图片直链/网页 URL 抓图片、下载到 `public/knowledge-images/`，再把 Markdown 图片引用追加到 `sourceExcerpt`。这不是完整路线 B：它不自动写 body、不做 OCR、不替作者判断图片里的事实。

**现在（卡片总数个位数）不该升级**，理由是算账：
- 路线 B 优化的是"出卡速度"。但你现在的瓶颈**不是出卡速度**，是"还不知道哪些卡值得做"——没有真实用户和缺口信号告诉你大家在问什么。
- 先把产能拉满，结果填一堆没人问的卡，是更大的浪费。

**真正该升级的触发条件**：当你手上积压了一批"明确知道该写、但人工一张张写不过来"的卡。那是个幸福的烦恼——到那时，出卡速度才真的成了瓶颈，路线 B 才回得了本。

在那之前，路线 A（AI 起草 + 你审）足够，而且**强制你逐张过目**，反而保住了质量。

## 7. 现在手上有什么工具

- **录入与管理**：`/admin/knowledge` —— 左边表单可手动新建/编辑卡，右边列表可搜索、按分区/核实状态筛选、归档。AI 写进库的 NEEDS_REVIEW 卡就在这里审。
- **缺口库**：`/admin/gaps` —— 用户问了但答不上来的问题汇集处，是"下一张该写什么"的信号源之一。
- **没解决反馈**：`/admin/knowledge/feedback` —— 用户对答案点了"这没解决我的问题"的汇集处，比空白缺口更精准地指出哪张卡该补角度。
- **NEEDS_REVIEW 的作用**：这个状态是人机分工的安全阀。AI 只能把卡写到这个状态，升 VERIFIED 永远是你的手动动作。所有问答检索会优先信任 VERIFIED 卡，全是 NEEDS_REVIEW 时 Agent 会倾向拒答而非乱答。

## 8. 架构演进备注（判断记录，不是当前待办）

> 这一节记的是 2026-05-31 一场长讨论想清楚的架构方向。**它不是现在要施工的东西**，
> 是给未来留的方向，免得几个月后忘了今天为什么这么决定、又走回头路。
> 一句话锚点：**事实能脱离情境去共享，经验脱离情境就变废话——所以事实分层、经验留卡。**

### 8.1 最终架构：两层 + 一个横切标注（不是三层）

```
底层  事实层 KnowledgeFact   —— 客观、会变、可共享、单一来源、能校验
上层  卡片层（=问题层）        —— 以问题为边界，body 是"带情境的经验"本身
横切  经验标签（最后再说）      —— 让经验能横向调取、归纳成"智慧"，但经验本体永不离卡
```

**为什么不是"事实层 / 经验层 / 问题层"三层**（曾认真考虑过，否掉了）：

- **经验不能单独成层**：事实（"五育平台=ndwy.nju.edu.cn"）抽出来单独放，意思不变；但经验（"别拖到大二"）一旦抽离它依附的情境和理由，就退化成正确的废话。美育卡和科光卡都说"别拖到大二"，但理由完全不同，抽成一条共享经验就把"因为……"杀死了。**所以经验就是卡片的血肉，把它抽走，卡片只剩一具引用骨架。**
- **"经验层"的结构化是假的**：事实层结构化能换来"单一来源+可校验"；经验层的"value"只能是一段话，等于把 body 换张表存，没换来任何好处，还多一层要维护。
- **问题层不用单独建**：它就是现在的卡片层，已经存在。
- **你真正要的"沉淀成智慧"**：智慧 = 从多条经验里归纳出的模式，靠的是一个**回顾视角**（经验标签把同主题经验横向串起来看），不是一个存储层。经验照旧长在各自卡里、带着情境。

### 8.2 事实层 schema 草案（已压测，待将来建表时直接用）

这版字段是用 14 种不同形状的真实硬事实压测过的，补了 `valueType` 才装得下"区间/上限/布尔/日期"。**记在这里是为了将来不用重想**——但它只是文档里的草案，**还没写进 `prisma/schema.prisma`**。

```prisma
model KnowledgeFact {
  id             String   @id @default(cuid())
  key            String   @unique   // 机器键，如 "labor_practice.credit"
  label          String              // 人话名，如 "劳动教育实践学分"
  valueType      String              // SINGLE / RANGE / MAX / BOOL / DATE
  value          String?             // 单值时用，如 "1"
  valueMin       String?             // 区间/上限下界
  valueMax       String?             // 区间/上限上界
  unit           String?             // "学分"/"小时"/null
  appliesGrade   String?             // "2025"；null=通用（如域名）
  appliesForward Boolean  @default(false) // 该级"及以后"是否适用
  sourceCardId   String?             // 溯源：这事实最初从哪张卡来
  note           String?
  createdAt      DateTime @default(now())
  updatedAt      DateTime @updatedAt
}
```

压测验证过的两个样例：
- 基础实践 10-15 小时 → `valueType=RANGE, valueMin=10, valueMax=15, unit=小时`
- 勤工助学 ≤5 小时 → `valueType=MAX, valueMax=5, unit=小时`

**事实层只装"单实体的客观值"。明确挡在门外的**（它们留在卡片 body 里）：
- 关系型（"青年科光要向老师报名"）、条件型（"志愿服务需团委负责才算时长"）、集合型（课号前缀那一串、悦读6模块）、跨事实校验（"5类学分相加=11"）。
- key-value 表结构上就表达不了"关系"，这不是缺陷——它正好当护栏，强制你别把经验/判断塞进事实层。

### 8.3 什么时候才真的去建这张表

**现在不建**（只有个位数卡片，事实冗余还不痛，建早了大概率设计错、还得改已有数据的表）。

**触发信号**：当你发现"**改一个政策，要翻好几张卡逐一改、还总怕漏**"——那一刻冗余之痛真实发作了，才是引入事实层的时机。在那之前，同一事实在多张卡里重复是可接受的（卡少，改得过来）。

**建表那天的具体操作**：先只建表 + 手动塞几条真实事实验证模型顺不顺手（纯加法，不碰 body、不碰检索、不建关联），不顺就删表重设计、零代价。让卡片真正"引用"事实、改检索把事实拼进答案，是更后面、要动 retrieval.ts / answer.ts 的重活，再单独评估。

### 8.4 你其实已经在打的"真基础"

不用现在多做任何事——你坚持的**溯源习惯**（每张卡都有 `sourceExcerpt` 逐字原文 + `sourceDescription` 来源）就是事实层的真地基。将来重构、要把散落事实收成原子时，每个事实都能回原始来源核对真伪。很多知识库恰恰死在"内容和来源脱钩、重构时没法验真"。**这条保持住就行，它已经在跑了。**

### 8.5 代码现状里埋的两条约定（2026-06-01，做 MiMo 起草前必看）

这两条不是待办，是代码现在已经成立的事实/约定，记下来免得将来踩坑或忘掉。

1. **AI 起草卡片，POST 时必须带 `draftedByAi: true`。** 创建接口 `src/app/api/knowledge/cards/route.ts` 的 POST 已经加了闸：带这个标记 → 无视传入的核实状态，**强制写成 `NEEDS_REVIEW`**；不带 → 尊重传入状态（人工录官方信息标 VERIFIED 是合理的）。所以 MiMo 那条线接进来时，**起草请求一定要带 `draftedByAi: true`**，否则"AI 卡必须人审"这条铁律的代码强制就形同虚设，退回到只靠自觉。注意闸只在 POST，不在 PATCH——PATCH 正是你人审完升 VERIFIED 的路径，不能拦。前端新建卡的默认核实状态也已从 VERIFIED 改成 NEEDS_REVIEW，"存疑"是默认动作。

2. **创建卡片没有去重检查。** POST 直接 create，不查"是否已有相同/相似 summary 的卡"，理论上能录两张几乎一样的卡。P0 量小、单人录，靠你自己把控即可，不值得现在写去重逻辑（过早建模）。**注意别和"缺口去重"记混**：`ask/route.ts:71` 那段 `findFirst(existingGap)` 是对**缺口**去重（同一个答不上来的问题已有 OPEN 缺口就复用、不新建），跟卡片无关。卡片这边确实没有任何查重。

## 9. 竞品对照得出的差距地图（2026-05-31 五份调研）

> 这一节是一次竞品调研的沉淀。拆了 5 个对象：Tiledesk、MaxKB、Dify、BARKPLUG V.2（论文）、ChatDKU/Duke BlueBot。
> **方法**：每条结论都要逐字原文引用 + 标 `[明示]/[推断]/[缺失]`，找不到就标缺失不编（这套纪律本身就是吃自己产品的狗粮）。
> **最大的元结论先写在前面**，免得被下面的清单带偏。

### 9.0 调研的元结论（比清单本身更重要）

1. **你和成熟产品/论文是"收敛"关系，不是"落后"关系。** 五份里反复确认你已经做对的：缺口做成一等数据库实体、未审卡不能单独支撑回答（拒答）、引用是结构化对象贯穿全链路、无出处不答、限官方源不联网。这些不是巧合，是多条独立路径走到同一个解——大概率是对的。
2. **它们的实现细节大半对你不适用，因为路线不同。** Tiledesk/MaxKB/Dify/BARKPLUG/ChatDKU 多是**向量检索 RAG**，你是**关键词打分检索**。所以它们的相似度阈值、chunk 大小、embedding 锁定、向量评测——对你不适用，别照搬。**你不是它们的简配版，是另一条路线。**
3. **边际收益已躺平。** 抓到第 3、4 个竞品时，"现在该动手"的新增洞察已接近 0，后面全是"再次确认你没错"。**继续抓竞品解锁不了下面的待办——只有真实用户能。**

### 9.1 差距地图：7 条真差距，按"现在能不能做"分层

调研里**真正"你还没做、且有价值"**的，是这 7 条。诚实分层——大多数被锁在"先有真实使用数据"后面，不是不做，是**现在做必错（没数据=凭空设计=又一个过早建模陷阱）**。

**A. 现在就能做（便宜、不依赖数据）**

1. **结构降噪规则**（来自 BARKPLUG 第6条）——✅ 已落地，见 [CARD-WORKFLOW.md §2 第2步补充](./CARD-WORKFLOW.md)。自动抓取入库前剔掉页脚/导航/备案号，防"带真 URL 的假溯源"。卡在 MiMo 自动起草的正前方。
2. **feedback 点踩 → 结构化成 gap candidate**（来自 BlueBot 第6条）——⬜ 未做。现状：`KnowledgeAnswerFeedback` 表只有 `note` 字段，是反馈日志；它和 `KnowledgeGap` 表**没接起来**。用户点踩不会自动变成"待补卡线索"。要做的是：点踩时把 `问题+回答+引用+评论` 打包进缺口工作流。**能做但要改代码动两张表，优先级低于"回去填卡"。**

**B. 该记成方向，但有前置条件（前置没到就别动手）**

3. **缺口回放测试**（MaxKB 第11条）——前置：库里有真实 gap 数据。审完一张卡，拿当初触发它的 `gap.originalQuestion`（逐字原话，schema 已存）回放，验证现在能否命中这张新卡、引对出处。把"降级会变哑"和"gap 存了原话"缝成闭环检查。
4. **两级缺口 / query rewrite**（ChatDKU 第2条）——前置：检索召回出现"问法没对上"的真实案例。现状 `retrieveKnowledgeCards` 是一次性检索，没命中就没命中。ChatDKU 在记缺口**前**先换个说法重检索一轮。**很多"缺口"其实是问法没对上、rewrite 一次就命中——你现在会把"问法没对上"误记成"知识缺失"。**
5. **失败记录存完整 trace**（ChatDKU 第12条 / BARKPLUG 第10条 / MaxKB 第3条收敛）——前置：你开始在意"为什么没答上"。现状 gap 只存 `originalQuestion`，中间链路没存。建议存全链：`原问 → 改写问 → 命中文档 → 证据充分性判断 → 最终回答/拒答`，否则事后分不清是检索错、判断错还是生成错（药不同：补卡/调阈值/改 prompt）。
6. **召回层 vs 回答层分离 + 问法别名**（Dify 第9/10/11 + MaxKB 第8 四条收敛）——前置：缺口库攒出一批真实学生问法。召回靠"学生会怎么问"（口语别名），回答靠"正式原文"。**别名该填什么值，由 `gap.originalQuestion` 真实数据反哺，不靠想象。** 现状召回靠作者写的 summary 关键词 + 仅 4 条硬编码 `RETRIEVAL_ALIASES`，学生口语命不中。
7. **hybrid retrieval 接口**（ChatDKU 第8条）——前置：卡多到关键词明显漏召回。最像你的 ChatDKU 用的是 `向量+关键词+rerank`，不是二选一。**修正一个之前的乐观叙事：你的"纯关键词"不是想清楚的架构优势，是 MVP 临时方案，迟早要补成 hybrid。** 现在卡少、关键词够用，不急，但别再当特色。

### 9.2 这张地图怎么用

- **A 类**做掉（第1条已做，第2条排在填卡之后）。
- **B 类**别现在动手——它们是**解锁地图**，不是待办。每条的前置条件到了，自己会浮现。逐条前置几乎都指向同一个燃料：**真实用户来问问题**（产生缺口、问法、点踩）。

## 10. 非竞品来源的借鉴（2026-06-01 第二轮，接 §9.2 的留口）

> 这一节填 §9.2 留的口。来源**不是竞品**，是更上游的设计思想：信息架构经典（Wikipedia、Stack Overflow、Zettelkasten）、RAG 工程文献（Anthropic）、以及一个缺口治理标杆（Intercom Fin，虽是产品但主业恰好是缺口管理，缺口这块密度最高）。
> **同一套纪律**：逐字引用 + URL + `[明示]/[推断]/[缺失]`。
> **这一轮的元结论**：竞品轮挖到的多是"再确认你没错"；非竞品轮挖到两类竞品给不了的东西——① 几十年沉淀的**治理原则**（维基的举证责任、SO 的重复即路标），② 你最弱环节（缺口治理）的**成熟拆法**（Intercom）。新增质量比竞品后几轮高，因为打在了设计思想层和你的软肋上。

### 10.1 信息架构经典（最肥的矿，竞品轮完全没有）

**① 维基百科：「可供查证，而非真实」+ 举证责任在添加者 —— 给你的 NEEDS_REVIEW 默认态背书，并补一个十年教训**

逐字原文（存档版 2008–2011 一致）：
> "The threshold for inclusion in Wikipedia is **verifiability, not truth**—whether readers can check that material added to Wikipedia has already been published by a reliable source, not whether editors think it is true." `[明示]` [archive 2011](http://web.archive.org/web/20110512011902/http:/en.wikipedia.org/wiki/Wikipedia:Verifiability)

映射：这是你 `verificationStatus` 的祖宗逻辑——**能不能被读者复查**（有 sourceExcerpt + URL）压倒**写卡人觉得对不对**。「不编造，留空 + NEEDS_REVIEW」= 维基的"举证责任在添加材料的人身上"。

**但有个反转值得抄**：维基后来（2012 RfC）把 "not truth" 从开头删了，因为被误读成"只要有来源，假的也能进"。现版改成软化措辞："verifiability means that people can check that facts or claims correspond to reliable sources" `[明示]` [current](https://en.m.wikipedia.beta.wmflabs.org/wiki/Wikipedia:Verifiability)。
→ 警示 `[推断，依据存档版与现版措辞差异]`：**「有溯源 ≠ 可发布」**。一张卡哪怕出处齐全，若内容过时（你的"旧版不录"），它满足 verifiability 却不满足 truth。维基踩了十年——你的 NEEDS_REVIEW 默认态 + 人工升 VERIFIED 正是这个补丁，**别让"有 URL"自动等于"可上线"**。

**② Stack Overflow：重复问题是路标，不是垃圾 —— 你可能正浪费 gap 资产**

逐字原文：
> "Duplicates act as sign posts, lighting the way to the canonical question. Your question used **different keywords, different language** to ask about the same concept, and now those keywords can be used to [find it]." `[明示]` [meta.SO](https://meta.stackoverflow.com/questions/265736/should-i-delete-my-question-if-it-is-marked-as-a-duplicate)
> "the same idea expressed in different words." `[明示]` [meta.SE](https://meta.stackexchange.com/questions/393411/duplicate-questions-with-no-accepted-answer)

映射 `[推断，由 SO 思想 + 你的 schema 推得]`：你 schema 有 `GapStatus.DUPLICATE` + `duplicateOf`，但大概率把重复 gap 当垃圾归档。SO 的设计是反的——**重复问题里那句"不同问法"本身是召回资产**。一个学生问"住哪"、另一个问"宿舍咋分配"，这两个 `normalizedQuestion` 应都挂到同一张 canonical 卡当检索别名。**这正是 §9.1 #4/#6（问法别名、召回层）的免费数据来源**：你库里攒的重复 gap 就是 `RETRIEVAL_ALIASES` 的素材（现在只有 4 条硬编码、全靠脑补），不用凭空想别名。

**③ Zettelkasten：原子性（一卡一念）—— 再确认你的卡模型，附一个边界判据**

逐字原文：
> "it is about putting **one idea and one idea only** on a note." `[明示]` [zettelkasten.de](https://zettelkasten.de/atomicity/guide/)
> "Each note captures a single idea. This makes notes **composable and reusable** across different contexts." `[明示]` [hahwul garden](https://examples.hwaro.hahwul.com/garden/notes/zettelkasten-method/)

映射：你的 summary（问题形状的检索把手）天然原子化，一卡一问。这条主要**再确认你的卡模型对**。边界判据值得回头自查：**如果一张卡的 body 塞了多个子问题**（"报到流程"混进"迁户口""领校园卡""医保"），summary 这个把手就糊了，打分被稀释。可执行判据：**一张卡的 summary 能不能用一句"怎么做 X"概括，不带"以及"**。

### 10.2 RAG 工程文献（主要再确认，一条新东西）

**Anthropic Contextual Retrieval —— 关键词检索不丢人，且你已在做"contextual"的人肉版**

逐字原文：
> "uses two sub-techniques: Contextual Embeddings and **Contextual BM25**." `[明示]` [Anthropic](https://anthropic.com/engineering/contextual-retrieval)

映射：BM25 = 关键词检索。Anthropic 的 SOTA 方案**明确保留关键词**、与向量并列。再次坐实 §9.1 #7 那句更正——纯关键词是 MVP 临时方案不假，但关键词本身不是"落后"，是混合检索的一半。
新的一点 `[推断]`：他们的"contextual"是在切片入库前给每个 chunk **预补一句上下文**，免得 chunk 成孤岛。你的 summary 字段（人写的、问题形状的把手）就是这个动作的人肉版——你已经在做，只是没意识到这跟 Anthropic 的 contextual prepending 是同一招。

### 10.3 Intercom Fin：缺口治理标杆（你最弱环节，这份密度最高）

> 来源是你 2026-06-01 提供的 17 条 Intercom 调研（逐字引用格式规范，#17 诚实标了 `[缺失]`）。下面只记**对照过 schema/retrieval.ts/answer.ts 后真有差异**的，按"能不能动手"分层。

**A. 真新、且已动手**

- **缺口分型 `gapType`（来自 Intercom content/data/action gap，#9）—— ✅ 已落地。** schema 已加 `GapType` enum（CONTENT / DATA / ACTION / POLICY_UNCLEAR）+ `KnowledgeGap.gapType` 字段（默认 CONTENT）+ 索引，migration `20260531164900_add_gap_type` 已应用。
  - 为什么值：你原来的 `GapStatus` 是**生命周期**（OPEN→HANDLED），不是**缺口性质**。没有 gapType，"我材料审到哪了"（DATA，补卡白搭）会和"转专业要啥条件"（CONTENT，补卡能救）混在一起，有人花半天给前者写卡——白费。加了类型，入口就把"写卡能救的"和"写卡白搭的"分开。
  - 为什么现在就做（不等数据）：**字段是埋点，埋点必须在数据产生之前埋好**。等攒起来再加，之前的缺口全是空值、得人肉回标。这是这批里唯一"必须现在做"的，已做完。
  - ⏳ 未接线：记缺口的代码还没往 gapType 写值（新缺口全默认 CONTENT），人审界面还不能选/改类型。**自动判类型的规则依赖真实缺口长相，属 §10.3 B 类，先别写。**

**B. 新、但撞 §9.0 的"volume 墙"（记下，别动）**

- **失败拆三类**（无答案 / 要人工 / 用户离开，#2）：思想对，但你只能观测"无答案"（retrieval.ts 已有 EMPTY/UNRELATED）+"差评"（feedback.note）。没人工通道、没存会话级交互 → 另两类观测不到。别加你填不了的字段。
- **失败根因 enum**（没召回对 / 用错卡 / 语气 / 长短 / 语言，#16）：是 §9.1 #2（feedback→gap）的具体化——把自由 `note` 换成可统计的原因码。但没真实差评量，enum 设计就是空想。
- **知识单元分用途分层**（长期政策卡 / 短期通知 snippet / 内部说明，#13）：和"年份必标/旧版不录"呼应，但这是在 `sourceType`（来源轴）外再加"用途轴"，与"一卡一念"原子性会打架。先别上。

**C. 第二信源强化 §9（非新增，但值得记"现在两个独立产品都这么做"）**

- **自动聚类 topic group（#1）+ SO 重复即路标（§10.1 ②）**：同一杠杆两个角度——重复 gap 的 `normalizedQuestion` 是 `RETRIEVAL_ALIASES` 的免费燃料。对应 §9.1 #4/#6。
- **回看当时命中哪些卡 + 排序（#5）**：= §9.1 #5 的 trace。现状 `KnowledgeQuestion.answer` 是 optional，GAP_RECORDED 的问题没存当时的失败回答和命中卡。**这条其实不太吃 volume，是个小代码改动**（记 gap 时把 retrieval 结果快照存下来），比 §9.1 归类的"有前置条件"更近一点。
- **批量阈值（min 10、首周后，#3）+ 拿人工答案反推缺什么（#4）**：背书 §9.0——"推荐补什么卡"要等量够了再做，别一条失败就误判。**是支持"回去填卡才是解锁键"的外部证据，不是新待办。**

**D. 纯背书你已做对的红线（回头确认即可）**

- **#12「不知道就转人工要检索后程序判断、不能只写 prompt 里」** = 你 retrieval.ts 的 `evaluateEvidence()`（检索后的程序级裁决，返回 EMPTY/UNRELATED/NEEDS_REVIEW），不是 answer.ts 那句系统提示词。**两年后回看别忘：判定放进 retrieval 而非 prompt，就是为了这个。**
- **#10 答不上→升级不伪装 resolved** = PRODUCT §5.2 无 fallback + retrieval.ts 硬拒。
- **#7 人审 accept/edit/reject 才上线** = NEEDS_REVIEW 默认态。
- **#15 反馈不直接训练** = 你压根没训练回路，自由 note → 人审，天然满足。
- **#11 低置信行为**：你主动**反对** Intercom 的"可尝试弱回答"，主张校园政策默认不答——**这个反驳是对的**。Intercom 是商业客服，答错=轻微烦躁；你的域答错=学生错过 deadline。风险等级不同，你该比它更紧。

**E. 边界（你自己标的 #17，确认）**

Intercom 只能拆**产品逻辑**，拆不了**工程实现**（闭源、无阈值/embedding/rerank）。工程实现仍要看 Dify/MaxKB/Onyx/Open WebUI 这类开源项目。`[缺失]` 标得对。
