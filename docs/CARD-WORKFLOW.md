# 知识卡片抓取与拆解工作流

> 这份文档回答一件事：**拿到一份原始来源，怎么把它变成合格的知识卡片入库。**
> 它是操作规范，不是产品论证。产品为什么这么设计，看 [PRODUCT-v1.md](./PRODUCT-v1.md)。
> 每一条规则都来自一次真实踩坑，写在这里是为了让下一个人不用再踩。

---

## 0. 一张卡片长什么样（字段速查）

| 字段 | 必填 | 说明 |
|---|---|---|
| `summary` 摘要 | 是 | 一句话，**写成"问题的形状"**，是检索的把手。≤200字 |
| `body` 正文 | 是 | 提炼后的知识点，**更高视角 + 提醒性**，不是原文复述。≤12000字 |
| `sourceExcerpt` 原文摘录 | 否 | 来源的逐字片段，可包含后台导入生成的本地图片引用，仅供用户核对溯源，**不参与问答检索、不喂给 LLM** |
| `sourceUrl` 来源链接 | 否 | 能跳转的原文 URL；来源是"问学长"这种没链接的就留空 |
| `sourceDescription` 来源说明 | 是 | 文字描述来源，**带上年份/版本**，如"南哪助手2.0（2025版）·02通识课" |
| `sourceType` 来源类型 | 是 | OFFICIAL/DOCUMENT/SENIOR/AUTHOR_EXPERIENCE/OTHER |
| `verificationStatus` 核实状态 | 是 | VERIFIED/UNVERIFIED/NEEDS_REVIEW |
| `domainTag` 领域标签 | 是 | 用细标签（如"选课规则"），别用泛标签（如"新生入学"），检索更准 |

## 1. 七步流程（照着做）

```
第0步  查库：动手前先搜库里有没有同主题/相邻主题的卡（按 summary 关键词搜）
第1步  抓取来源原文（Tavily extract / 手动粘贴）
第2步  砍掉不该录的：①结构噪声（导航/页脚/相关推荐/备案号等非正文）②旧版本 ③过时信息 ④与本心无关的内容
第3步  按"一个能被单独提问的知识点 = 一张卡"切颗粒（颗粒怎么定，见 §1.5）
第4步  每张卡写 summary（问题形状）+ body（高视角提炼）
第5步  填 sourceExcerpt（逐字原文）+ 来源元数据（含年份）
第6步  定 verificationStatus：有残缺/存疑的标 NEEDS_REVIEW
第7步  入库（手动表单 /admin/knowledge，或批量 seed 脚本）
```

### 第5步补充：原文图片怎么收

如果来源里的图片本身承载信息（表格、流程图、数据图、操作截图、重要示意图），后台 `/admin/knowledge` 可以在"从 URL 导入原文图片"里粘贴图片直链或网页 URL。

- 图片直链会直接下载并追加到 `sourceExcerpt`。
- 网页 URL 会先列出候选图片，人工只勾选真正有信息量的图片。
- 系统会把图片存到 `public/knowledge-images/`，并在 `sourceExcerpt` 里追加 `![原文图片](...)` 形式的本地图片引用。
- **图片强制放 sourceExcerpt**：所有图片一律进 `sourceExcerpt`，不进 `body`。body 只写文字。
- 这仍然只是溯源展示：如果图里的事实要参与回答，必须把核实后的关键信息写进 `body`；本功能不做 OCR，也不自动总结图片。

> ⚠️ 对 AI / 自动化执行者：用 `fetch-yuque.ts` 抓取页面后，**必须用 `--download-images=<编号>` 参数下载图片**，然后用本地路径 `![图片名](/knowledge-images/xxx.png)` 引用。**不能直接用语雀 CDN 链接**（`https://cdn.nlark.com/...`），否则图片在后台无法渲染。这是 2026-06-02 宿舍卡片踩出的教训。

### 第5步补充：原文超链接怎么收

抓取网页时必须提取超链接（`<a>` 标签的 href），在 `sourceExcerpt` 中保留 Markdown 格式的链接 `[text](url)`。

- 抓取脚本（`fetch-yuque.ts`）会自动提取超链接并输出。
- 写卡时把原文中的超链接补充到 `sourceExcerpt` 和 `body` 中。
- 超链接是溯源的重要依据，丢失链接会降低卡片可信度。

### 第0步为什么单列：防止悄悄造重复卡

这一步是真实踩出来的：曾经差点为一张"劳动教育导引图"新建卡，结果库里早有两张劳动教育的卡，撞了就是检索冲突。所以动手前**必须先查**。

**查库不是看一眼就算，要落成一个可检查的动作**：起草前先输出一句明确的现状声明——

```
现状：库里已有《X》《Y》两张相关卡 ／ 库里没有相关卡
```

- 撞到同主题 → **改做补全或订正那张已有卡，不要新建**（怎么订正见 §3.5 真实案例）。
- 确实没有 → 声明"无相关卡"，再往下走。

> ⚠️ 对 AI / 自动化执行者：此步依赖你**具备查库能力**（能写查询脚本、跑起来读结果）。
> 首次把工作流交接给新执行者（如多模态模型）时，**必须先实测它会不会查库**，
> 别默认它会。它不会查的话，这一关要由作者在审核侧补上（见
> [使用与迭代.md §3 第五关](./卡片工作流-使用与迭代.md)）。

## 1.5 一张卡装多少：按"提问的形状"切，不按"主题"合

第3步切颗粒时，最容易纠结的就是"这些内容该合一张还是分几张"。一把尺解决：

- **合**：同一个问题的答案，不该散在多张卡。
  例：实践课"怎么报名"和"时长怎么算"，是同一个问题的两面 → 合一张。
  （反面教训：当初把时长丢给一张图、报名写正文，图 OCR 挂了就缺了半个答案，见 §3.5）
- **分**：会被分别提问的两件事，不该挤在一张卡。
  例："理论课怎么修"和"实践课怎么修"是两次独立提问 → 分两张。

**尺子是"用户会不会在一次提问里同时想知道"，不是"是不是同属一个主题"。**

- 容量（body ≤ 12000字）是上限，但**没触到上限，也别因为"都属于劳动教育"就硬合**。
- 卡片是"问答单元"，不是"百科词条"。一张卡塞成一篇说明文，用户问任何细节都被一大坨淹没，反而难精准回答。

> 注：同一个客观事实（学分数、域名等）在多张卡里各自重复，**现阶段是可接受的**。
> 别为了消除重复去硬合卡——消除事实冗余是"事实层"将来的事，不是切颗粒时该操心的。
> 详见 [使用与迭代.md §8 架构演进备注](./卡片工作流-使用与迭代.md)。

## 2. 三条铁律（违反就是在毁产品）

1. **不编造**：抓取乱码、图片识别失败、原文没写的，**留空 + 标 NEEDS_REVIEW**，绝不靠猜补全。给残缺内容盖"已核实"章，是这个产品最该死的错。
   - **延伸：来源外不扩展**。来源只给了列表/事实，不等于你可以补制度解释、专业介绍、流程说明。"强基计划是国家基础学科拔尖学生培养计划""匡亚明学院是南京大学基础学科拔尖人才培养的特区学院"——来源没写这些，你写了就是编造。来源是纯列表时，末尾可加使用提醒，但不加制度解释。
2. **旧版不录**：来源里有多版本（如2024级/2025级方案），只录**当前级**，旧版直接丢。两版都录会让检索命中冲突卡片，用户更懵。注意区分：这是指**政策/规则的旧版**（如旧版培养方案、旧版分流规则），不是指**历史数据**（如18级保研率、2019届就业数据）——后者可以作为参考保留，但必须标注时效（见§3.11）。
3. **年份必标**：`sourceDescription` 必须带年份/版本（如"南哪助手·金陵通交通卡（2025版）"）。`body` 结尾**不要**加"（信息适用于X级方案）"这类标注，除非用户明确要求。

### 第2步补充：自动抓取要先剔"结构噪声"（手动粘贴一般没这问题）

第2步的"砍"分两层，别只砍一层：

- **语义噪声**（旧版本、过时、跑题）——人读得出来，原本就在砍。
- **结构噪声**（导航栏、页脚、"相关推荐/上一篇/下一篇"、备案号、侧边栏）——**自动抓取（Tavily/网页）才会混进来，手动粘贴时人眼自然滤掉了**。

为什么单独拎出来警告：结构噪声里最危险的是**页脚/侧栏的文号、日期、链接**。它们带着真实出处，看起来权威，一旦被起草者当成正文里的政策依据收进卡，就造出"**带着真 URL 的假溯源**"——这比单纯抓错更隐蔽，因为它骗得过"有没有出处"这一关。

> ⚠️ 对 AI / 自动化执行者：用 Tavily/网页自动起草时，**入库前必须先剔掉结构噪声**，尤其别把页脚/侧栏的文号、日期当成正文政策依据。手动粘贴的来源一般无需此步。这条来自 BARKPLUG V.2 的 "manual div selection / semi-automatic" 取舍——成熟校园 RAG 也是在入库前人工降噪，不让噪声进库变成有出处的错误答案。

## 2.5 一条正向硬指标：每张卡至少一句"原文里没有的话"

三铁律拦的是**错误**（编造、旧版、缺年份），但拦不住**平庸**——一张字字属实、却只是把原文誊了一遍的卡，能通过三铁律，却没有存在的理由。

**这个产品凭什么比让用户直接去读语雀/官网强？** 唯一的理由是：卡里有过来人的判断，是原文给不了的东西。如果一张卡删掉之后，用户看原文能得到完全一样的信息，那这张卡就是套壳，是在稀释产品。

所以加一条正向硬指标，跟三铁律一样是红线：

> **每张卡的 body，至少要有一句原文里没有的话。**

这"一句话"可以是：
- **经验排序**：原文平铺三个模块，你点出"基础实践那 10-15 小时才是主力，别指望勤工助学凑满"。
- **踩坑提醒**：原文写"志愿服务可计时长"，你补"必须是团委负责的项目，别的不认"。
- **时机建议**：原文写"大一至大三均可"，你补"趁大一有空先占掉，拖到后面没空"。
- **反直觉澄清**：原文写"自动置入课表"，你补"别因为不是自己选的就忘了它存在"。

**检验法**：写完 body，问自己——"把这张卡删了，用户看原文会漏掉什么？"答得出来，这张卡才立得住；答"不会漏什么"，退回重写，哪怕它一个字都没错。

> ⚠️ 这条指标和"不编造"不冲突：补的是**判断和经验**（怎么安排、哪里坑、什么时机），不是**事实**（学分、学时、报名地址）。事实只能来自来源，判断来自过来人。分不清这两者，就会把"编造"伪装成"洞察"——那是更危险的错。

## 2.6 来源不确定的，不能写成确定的

来源写「不一定」「需要等通知」「可能有变化」时，body 里**不能改成确定时间或确定结论**，必须保留不确定性表达。

这是「不编造」铁律的延伸——编造不只是捏造事实，也包括把不确定的信息写成确定的。

**反面教训**：来源写「新生不一样需要等具体通知」，AI 初稿写成「新生退补选时间为 X 月 X 日」——信息源是"等通知"，AI 却给出了确定时间，违反铁律。

**检验法**：写完 body，对照 sourceExcerpt 检查：原文用「可能」「不一定」「等通知」的地方，body 里是不是也保留了不确定性？没有就改回来。

## 3. 正文的语气：像过来人，不像班主任

这是最容易做错、也最伤产品的一点。

**产品的本心是"过来人把经验留给后来人"。** 过来人说话的样子是"这块我当时差点栽了，提醒你一句"，**不是**班主任站讲台上划重点、念祈使句。一个新生读到满屏的方括号标签和"务必""切记""别漏掉"，感受到的是被管教，不是被帮。这跟产品想给的体验是反的。

### 三条语气规则

1. **不要套结构化标签**：别用「【一句话】【关键动作】【两个坑】【提醒】」这种格式。信息分段用自然换行就好，让它读起来像人在讲话。
2. **少用祈使句**：把"务必把握""切记别拖""一定要加群"换成陈述加轻提醒，如"这个会自动进课表，容易被忽略"。
3. **高视角但平视**：正文要比原文多给东西（过来人的判断、容易踩的点、为什么这么安排），但姿态是平视的同伴，不是俯视的老师。

### 正反示例

**爹味/机械（不要这样）**
```
【一句话】劳动教育理论1学分，大一第一学期会自动塞进课表。
【关键动作】务必主动加课程QQ群！考试千万别忘，忘考必挂！
【提醒】这是硬性要求，切记不要漏掉。
```
问题：方括号标签像教学PPT；务必、千万、切记三连祈使句像训话；新生读着有压迫感。

**过来人平视（这样写）**
```
劳动教育理论1学分，大一第一学期会自动塞进你的课表，本质是刷网课。
别因为不是自己主动选的，就忘了它存在。

开学后要主动做的事：课程分3个班，得加对应的课程QQ群（在哪个班看课表），
考试别忘记，忘考会挂。万一挂了也别慌，春季学期可以重修，不是一锤子买卖。
```
区别：信息量一样，但去掉了标签和祈使连发；"别因为…就忘了它""不是一锤子买卖"是过来人语气，不是命令。

**再看一个对比（科学之光）**

| 维度 | 教学腔（不要） | 过来人（这样） |
|---|---|---|
| 开头 | 【课程简介】科学之光为1学分必修通识课 | 科学之光1学分，分大科光和青年科光两种 |
| 给建议 | 务必在大一修完，切勿拖延！ | 和美育一样，没选上可以大一下补，但别拖到大二 |
| 整体感受 | 像课程大纲 | 像学长跟你唠两句 |

### 一句话检验法

写完一张卡的正文，读一遍问自己：**这像学长私下跟我说的话，还是像辅导员发的通知？** 如果是后者，重写。

### Markdown 结构化（§3 补充）

语气规则反对的是**机械的结构化标签**（【一句话】【关键动作】），不反对用 Markdown 提升可读性。恰恰相反，大段纯文字会把关键信息淹没。

**建议用 Markdown 的**：
- 标题（##、###）区分信息层级
- 列表（-、1.）呈现并列信息
- 加粗（**关键词**）突出限定词、数据、警告

**仍要避免的**：
- 【】方括号标签
- 连续祈使句（务必/切记/千万）

**检验法**：写完 body，问自己——「这段话有没有一眼能看到的关键信息？」答"没有"，考虑用列表或加粗组织。

## 3.1 行动步骤式写法（「怎么做」类卡优先给步骤）

对于「怎么查/怎么做/怎么选」类问题，正文应**优先给出可执行步骤**，再给背景说明。

新生需要的是「我现在该做什么」，不是「这个制度是怎么设计的」。先给步骤让他能行动，再解释为什么这么设计。

**正反示例**

**说明文式（不要）**
```
保研课是根据各专业推免细则确定的，通常包含专业核心课程和
部分通识课。学生可以通过教务系统查询推免细则文件...
```

**行动步骤式（这样写）**
```
查保研课最短路径：

1. 登录教务系统 → 个人中心 → 推免细则
2. 找到「推免生学业成绩计算办法」，里面会列出哪些课计入保研排名
3. 把这些课标注出来，选课前先确认它们是否在本学期开设

查不到怎么办？公示期结束后文件会被删除，可以找辅导员或教务员要往年版本参考。
```

**检验法**：写完「怎么做」类卡的 body，问自己——「新生看完能直接开始第一步吗？」答"不能"，说明步骤还不够具体或埋在背景说明里了。

## 3.2 风险前置：最大坑用小标题或首段突出

有「最大坑」「最关键的点」时，**用小标题或首段突出**，不要埋在正文中间。

新生没有耐心通读全文，如果关键风险埋在第三段中间，大概率看不到。用小标题（如「最大的坑」「最容易踩的坑」）或首段直接点出，让他一眼就能注意到。

**正反示例**

**埋在中间（不要）**
```
课程替代分为校级、院级和毕业替代三种。校级替代指...
毕业替代需注意，如果替代课程与原课程学分不同，可能会
影响毕业学分计算。另外，有些学院可能不承认...
```

**风险前置（这样写）**
```
课程替代分为校级、院级和毕业替代三种。

最大的坑：毕业替代需注意——如果替代课程与原课程学分不同，
可能会被直接判定为缺课。申请前务必找教务员确认。
```

**检验法**：写完 body，问自己——「这张卡最大的风险是什么？它在前三行能看到吗？」答"看不到"，调整位置。

## 3.3 时机建议要具体到真实场景

不要只说「提前确认」「选课前注意」，要**具体到「选课/订票/安排回家时间前」等真实场景**。

「提前确认」是抽象建议，新生不知道提前到什么时候、在什么动作之前确认。具体到真实场景（订票、选课、安排回家），他才能把建议落地。

**正反示例**

**抽象（不要）**
```
选课前先确认自己专业有没有强制暑期课程。
```

**具体（这样写）**
```
订票或安排回家时间前，先确认自己专业有没有强制安排，
以及所选课程实际要上几天——别票买好了才发现走不了。
```

**检验法**：写完时机建议，问自己——「新生看到这句话，知道在哪个动作之前做这件事吗？」答"不知道"，说明场景还不够具体。

---

## 3.4 来源内部冲突的处理

来源不同位置对同一信息有矛盾表述时，**不要自行判断对错**，也不要只挑一个版本写。

**处理方式**：
1. 在 body 中明确标注「来源表述不一致」
2. 将两处原文都保留在 sourceExcerpt 中
3. body 中写「需以教务系统/院系通知为准」

**反面教训**：英语课卡片一处写「四层次需要上四个学期」，另一处写「别的四层次同学只需要上两学期」，初稿只取了一个版本，造成两卡之间自相矛盾。

**检验法**：写完 body，对照 sourceExcerpt 检查——原文中有没有对同一事项的不同说法？body 里有没有只取了一个版本？

---

## 3.5 限定词不能省，绝对化表述要加限定

原文的限定词（「有些」「部分」「很多」「一般来说」）必须保留，省掉就从「特定情况」变成「普遍规则」。原文的绝对化表述（「没有任何意义」「完全不」等）写入卡片时，应加「来源提醒」「不建议」等限定词。

**反面教训 1**：原文写「有些专业课是教务员指定班级且不允许换班」，初稿省掉「有些」写成「如果教务员指定班级就不要自行换班」，把特定情况变成了普遍规则。

**反面教训 2**：原文写「准入/准出没有任何意义」，初稿照搬。新生看到「没有任何意义」比「容易造成误解，不建议作为依据」更不安。

**检验法**：写完 body，对照 sourceExcerpt 检查——原文的「有些」「部分」「一般来说」等限定词是否都保留了？有没有把特定情况写成普遍规则？

### 制度用语的精确性

制度文件的措辞有精确含义。「可以给予XX处分」≠「会被XX处分」——前者是裁量空间，后者是确定后果。照搬时省掉「可以」，就把"有权这样做"变成了"一定会这样做"，对用户判断影响很大。

**反面教训**：原文写「可以给予开除学籍处分」，初稿写成「会被开除学籍」，省掉了裁量空间。制度用语的精确性直接影响用户判断。

**检验法**：写完 body，检查制度类措辞——原文的「可以」「应当」「可以给予」有没有被省略或替换成更确定的表述？

---

## 3.6 图片数据的引用

图片中的具体数字（学分、门数、时间等）写入 body 时需谨慎：sourceExcerpt 中的图片引用无法让审稿人直接核验数字内容。

**处理方式**：
1. 图片中的数字如果可能因专业/年级不同而异，优先用「以培养方案为准」的通用表述
2. 如果必须写具体数字，在 body 中明确标注「从图片中可以看到」或类似出处提示
3. 不要假设图片中的数据适用于所有情况（如大气科学专业的数据不等于所有专业都是这个数字）

**检验法**：body 中有具体数字时，问自己——这个数字是所有专业通用的，还是特定来源的？如果是后者，要不要加限定？

---

## 3.7 标题/summary 应反映来源实际覆盖范围

标题（summary）应精确反映来源能回答的问题范围，不要因为卡片名好听就扩大承诺。来源只讲了某一个方面，标题就不该暗示会解释所有方面。

**反面教训**：来源只讲了「专业选修课不算学位学分绩」，原标题写「学分绩怎么算」暗示会解释所有学分绩计算规则。改为「专业选修课影响哪类学分绩？」后准确反映了来源范围。

**检验法**：写完 summary，问自己——用户看到这个标题，会期望卡片回答什么问题？来源真的能回答这个问题的所有方面吗？答「不能」→ 改标题。

## 3.8 一个真实案例：用新来源补全旧卡的"待核实"窟窿

这个案例值得记下来，因为它示范了铁律落地后**怎么收尾**——不编造留下的坑，后来怎么填。

**背景**：劳动教育实践卡当初入库时，原文里那张"劳动教育学习导引图"被 OCR 识别成乱码。按"不编造"铁律，作者没有瞎补时长数字，而是在 body 里**明写了一段占位**：

```
待核实：实践时长的具体构成（基础、学科、综合实践模块各需几小时，勤工助学
与志愿服务各自上限）原文档是用一张导引图给的，本卡抓取时图被识别成乱码，
暂未收录，需对照原图补全。
```

这段占位是**正确做法**：它没有给残缺内容盖"已核实"章，而是诚实标注缺口、等真实来源。

**收尾**：后来用能读图的多模态模型读出了那张导引图，于是：
1. 把"待核实"那段**替换**成图里读出的真实数据（三个模块的小时数、各自上限）。
2. 顺手发现年份标窄了——卡标"2025级"，但图明确写"适用于2021级及以后"，于是 body 里年份改成更准的"2021级及以后"。
3. 清掉 sourceExcerpt 结尾那句"（图OCR乱码未收录）"，因为图的内容已经收录了。
4. **状态从 VERIFIED 降回 NEEDS_REVIEW**——因为新数字是 AI 从图读的，还没经作者确认，符合"AI 出草稿、人审"的分工。

**这个案例的三个教训**：
- 抓取失败时，**留诚实的占位**比硬补强得多——它给未来留了明确的填坑指引。
- 一张 VERIFIED 卡里**不该带"待核实"字样**，这本身不自洽；发现了就是该收尾的信号。
- AI 读出的新数据先标 NEEDS_REVIEW，**升回 VERIFIED 是作者的权力**，不是 AI 自作主张。

## 3.9 决策类卡要先给结论和判断空间，再给制度细节

「能不能进」「要不要报」「值不值得修」类问题，新生最先想知道的是**「我有没有操作空间」**，不是这个制度是怎么设计的。如果第一屏全是条件清单和制度条款，新生不知道哪些跟自己有关、哪些不用管。

**卡结构建议**（适用于决策类卡）：

1. **一句话结论**：先告诉用户这件事对他有没有操作空间（如「多数人不需要参加，但部分专业有强制课」）
2. **你能不能申请/进入**：用来源中的事实做判断
3. **具体有哪些选择/什么时候关注**
4. **关键影响**（保研、考研、法考等冲突）：涉及「如果做了A，就不能做B」的决策冲突要单独突出
5. **坑点提醒**：最重要的踩坑信息前置，不埋在正文中间
6. **适合谁/不适合谁**：基于来源事实给判断

**正反示例**

**制度说明式（不要）**
```
双学士学位是经教育部审批设立的双学士学位复合型人才培养项目。
我校现有6个双学位班：计算机与金融工程、软件工程与经济学...
（大段制度描述）
```

**决策卡式（这样写）**
```
先说结论：如果你高考填志愿时报了双学位班，你已经在里面了；
如果没报，本科期间没有入口再进。

现有6个双学位班及方向：...
最大的坑：辅修的法学不允许参加法考——不要把「学信网有法学学位
信息」等同于具备法考资格。
```

**检验法**：写完决策类卡，问自己——「新生看完第一段，知不知道自己有没有操作空间？」答"不知道"，说明结论前置不够。

## 3.10 审慎表达：非常规路径不写成推荐，事实和评价分层

### 审慎表达

来源提到非常规路径（如退学重考、放弃保研转考等）时，body 里**不能写成推荐策略**，要加边界提示。读者可能把经验描述当成操作建议，但这类路径涉及个人情况和当年政策，一张卡承担不了这个决策责任。

**反面教训**：「退学仍算应届生身份」被写成推荐语气，缺风险边界。修改为：「这类路径涉及个人升学安排和当年政策理解，不能只靠这张卡决定」。

**检验法**：写完 body，问自己——「新生读完会不会以为这张卡在推荐我这么做？」答"会"，加边界。

### 事实与评价分层

来源中「客观事实」（学信网怎么显示、证书上写什么）和「来源评价」（含金量很高、性价比不高）应该分开写：**先写事实，再写评价**。混在一起会让读者把评价当成事实，或者把事实当成主观判断。

**正反示例**

**混写（不要）**
```
学信网上会显示法学学士学位信息，含金量很高。
```

**分层（这样写）**
```
学信网学历信息详情页会标注「辅修法学」，学位信息有两条
（主修+辅修）。来源认为辅修的含金量很高，求职时有认可度。
```

**检验法**：写完 body，问自己——「哪些是来源里的客观事实，哪些是来源或作者的评价？读者能分清吗？」答"分不清"，拆开写。

---

## 3.11 宣传化表述转中性 + 历史数据必须标注时效

### 宣传化表述转中性

学院介绍类来源常包含宣传口径（「中国最好的大气学院」「要有自信」「颜值很高」）。知识卡片应使用**中性、事实型表达**，删除主观评价和宣传化措辞。

**处理方式**：
1. 「最好的XX」→ 改为客观表述（如「在历次学科评估中表现靠前」）
2. 「要有自信」→ 删除，或改为事实支撑（如「学科排名靠前」）
3. 「颜值很高」→ 删除，或改为中性描述（如「男女比例相对均衡」）
4. 「关系亲密宛如家人」→ 改为事实描述（如「学院规模较小，同学之间上课和学习交集较多」）
5. 「奖品丰厚到你会怀疑学院是不是嫌钱太多花不掉」→ 删除

**检验法**：写完 body，问自己——「这段话放在学院官网招生简章里合适吗？」答"合适"，说明宣传味太重，改中性。

### 历史年份数据必须标注时效

来源中引用旧年份数据（如18级、19级、2019届）时，body 里**必须标注数据的时间属性**，防止新生误以为是当前数据。

**处理方式**：
1. 旧年份数据前加「来源以XX级为例：」
2. 结尾加「该数据仅作参考，实际名额/比例以当年学校通知为准」
3. 不要删除旧数据——它是有价值的参考，但必须标注时效

**反面教训**：保研率数据用18级数据但未标注年份，新生可能误以为是当前保研率。

**检验法**：写完含数字的 body，问自己——「这个数字是哪一年的？读者能看出来吗？」答"看不出来"，加标注。

## 3.12 相似名称课程要区分位置

同一来源中可能有名称相似但性质不同的课程（如"大学物理"和"普通物理学"），不能想当然认为是同一门课。

**处理方式**：
1. 根据它们在课程结构中的位置（必修 vs 选修池）分别处理
2. 如果原文没有明确说明，需标注"需以培养方案为准"

**反面教训**：化生大类卡中把"大学物理"和"普通物理学"都列为必修课，但实际"大学物理"在选修课池里，"普通物理学"才是必修。

**检验法**：写完课程清单，问自己——「名称相似的课程是不是同一门课？它们在课程结构中的位置是否一样？」

---

## 3.13 术语要精确

制度文件的措辞有精确含义，不能随意替换。数据描述也要用准确的术语。

**处理方式**：
1. "报录比"≠"录取率"：报录比通常指报名人数÷录取人数，录取率指录取人数÷报名人数
2. 制度用语的精确性（见§3.5）：「可以给予XX处分」≠「会被XX处分」
3. 如果不确定术语的精确含义，保留原文表述

**反面教训**：转电子卡中用"报录比"但实际数据是"接收人数÷报名人数"，严格来说是"接收率/录取率"。

**检验法**：写完 body，问自己——「数据描述的术语是否精确？制度用语是否被替换了含义？」

---

## 3.14 课程卡片先列清单再评价

涉及课程的卡片，应先写课程清单，再写经验评价，最后写选课提醒。

**结构建议**：
1. **课程清单**：先列出必修课、选修课池，明确哪些是必修、哪些是选修
2. **经验评价**：再写来源对各课程的评价（难度、老师、给分等）
3. **选课提醒**：最后写选课注意事项（学分要求、二选一等）

**反面教训**：化生大类卡中课程清单和评价混在一起，新生难以快速找到自己需要的课程信息。

**检验法**：写完课程卡，问自己——「新生能不能先快速找到课程清单，再看经验评价？」

## 3.15 Q&A类来源切法

Q&A格式的来源（如"转CS指北"中的Q&A部分），应按问题分散到对应主题卡中，不单独成卡。

**处理方式**：
1. 把Q&A中的每个问题归入对应的主题卡（如"数学课怎么选"归入数学课卡）
2. 如果某个问题无关紧要（如"其他问题在群里问"），可以跳过
3. 写卡后逐条核对Q&A中的每个问题是否已覆盖

**反面教训**：转CS指北的Q&A部分有8个问题，初稿未覆盖，需要补充到对应卡片中。

**检验法**：写完Q&A类来源，问自己——「Q&A中的每个问题是否都已归入某张主题卡？」

## 3.16 来源边界要清晰

每张卡正文只使用本卡 sourceExcerpt 支撑的信息，跨卡引用需明确标注。

**处理方式**：
1. 每张卡的 body 只使用本卡 sourceExcerpt 支撑的信息
2. 如果需要引用其他卡的内容，写"分流名额需要看分流卡"这样的跨卡引导
3. 不要从其他卡的来源中取信息塞进当前卡

**反面教训**：化生大类卡中卡片1混入了分流卡来源的信息，卡片4使用了生科段来源的信息。

**检验法**：写完 body，问自己——「这张卡里的信息，是否都能在本卡的 sourceExcerpt 中找到出处？」

## 3.17 来源外不扩展的三种常见形态

§2 铁律第1条「不编造」已经覆盖，但本批次暴露了三种具体形态，值得单独列出：

1. **补组织架构**：来源只说了某专业在某学院下，AI 就补了「该学院是XX领域的重点学院」。院系归属、校区归属等组织信息，sourceExcerpt 中没有就不能写。
2. **补具体数字**：重修费用90元/学分、缓考成绩折算90%、缓修门数上限等具体政策数字，来源没给就不能写。不能从「可以重修」推断出「重修费用XX元」。
3. **补操作路径**：ehall.nju.edu.cn → 本科生服务 → 教务管理 这类具体系统路径，来源没给就不能写。改为「按教务系统提示操作，入口以当年通知为准」。

**检验法**：写完 body，问自己——「这条信息在 sourceExcerpt 里能找到原文或明确推导吗？」答「找不到」→ 删除或改为「需以当年通知/辅导员/教务答复为准」。

## 3.18 新生卡必须有可执行建议

本批次14个任务中，10个的第一版都缺少「新生现在该做什么」——GPT 审查的核心改进方向。

**问题**：卡片只告诉新生「这是什么」，没有告诉「我该怎么行动」。新生读完知道了一个制度，但不知道自己该在什么时候、做什么动作。

**要求**：每张卡至少包含以下一项：
- **行动步骤**：「第一步做X，第二步做Y」（适用于「怎么做」类卡）
- **判断框架**：「如果你的情况是A，选X；如果是B，选Y」（适用于「要不要」类卡）
- **时机提醒**：「这件事要在XX之前做」（适用于有时限要求的卡）
- **不确定时问谁**：「不确定就问辅导员/教务员」（适用于规则可能变化的卡）

**检验法**：写完 body，问自己——「新生看完能直接开始第一步吗？」答「不能」→ 补充可执行建议。

## 3.19 踩坑提醒前置，不埋在结尾

本批次4个任务的初稿把踩坑提醒放在总结段或正文中间，新生大概率看不到。

**要求**：踩坑提醒应在正文前半段出现（小标题或首段），不要等读者读完全文才发现。§3.2 已有「风险前置」规则，这条是它的延伸——踩坑提醒也属于高风险信息，适用同一原则。

**检验法**：写完 body，问自己——「这张卡最大的坑是什么？它在前三行能看到吗？」答「看不到」→ 调整位置。

## 3.20 个别年级经验不能写成普遍结论

本批次3个任务（003/006/007）把「22级考前复习一天」「22级智科超容量未淘汰」等个别年级经验写成了普遍结论。

**处理方式**：
1. 个别年级/个别案例的经验，必须标注「按XX级经验」或「来源以XX级为例」
2. 不能从个别年级经验推断「给分友好」「不会淘汰」「可以摆烂」等普遍结论
3. 保留经验的参考价值，但明确其局限性

**反面教训**：「22级考前复习一天」被写成「给分友好，可以摆烂」——前者是个别经验，后者是错误推断。

**检验法**：写完含数字或评价的 body，问自己——「这个结论是所有年级通用的，还是特定年级的？」答「特定年级」→ 加限定。

## 4. 给 AI 助手/自动化执行者的额外提示

- 你**不掌握** NJU 的真实事实，不许凭训练知识生成卡片内容。内容只能来自用户提供的来源。
- 拆解时主动盯"范围蔓延"：来源里有大量内容时，只挑**高频、稳定**的知识点做卡，不要追求"全导进来"（这会摊薄、且把本产品做成来源的套壳）。
- 批量入库用 seed 脚本时注意：正文/摘录里**不要用 ASCII 双引号**做强调（会截断 TS 字符串），需要强调用直角引号「」或直接不加。
- 默认语气会滑回"结构化教学腔"，写完务必用上面的"一句话检验法"自查。
- **改后全文搜索**：修改文字（如改错别字）后，用搜索功能确认无遗漏。
- **不标年份**：body 结尾不要加"（信息适用于X级方案）"，除非用户明确要求。
- **原文确认**：入库前必须展示 sourceExcerpt + sourceUrl 给用户确认，不能直接入库。
- **总结要到位**：AI 总结原文时容易丢失关键细节（如时间限制、原因解释、限定词）。写卡后需人工核对原文中的因果关系和限定条件，不要含糊带过。特别是原文中明确列出的「优点」「缺点」「条件」「步骤」等结构化内容，body 里必须逐条覆盖，不能漏。
- **写卡前逐项列出**：写卡前先列出原文所有条目（bullet points），写卡后逐条核对是否覆盖。不能凭印象写卡，否则会遗漏独立类别（如洗浴）或跳过简略条目（如「苏浙体育场等」）。
- **结构跳跃要检查**：原文有清晰结构（如10个报销类型、多个并列section）时，不能只关注头尾或最大的两块，要检查中间是否有独立小节被遗漏。常见陷阱：只抓「门诊大病」和「异地就医」，跳过中间的「门诊精神病」「门诊艾滋病」「不报销情况」。
- **来源为空时跳过**：抓取后发现 word_count=0 或页面仅显示标题（语雀 status: 0 通常表示未发布），应直接跳过、不创建卡片，输出空 cards.json。不要浪费任务配额去尝试多种抓取方式——来源没内容就是没内容。

### 交接给新执行者时，必须知道的环境事实

把工作流交给一个新会话/新模型（如多模态模型代笔）时，下面这些它看不到、得明确告诉它：

- **入库走 Prisma 直接写库**，不要调 `POST /api/knowledge/cards`——那个接口有登录态鉴权（`requireKnowledgeAuthor`），脚本拿不到 session。
- **`createdById` 不要抄死值**：用脚本自己查管理员，别把某个用户主键写进文档/脚本（换库就失效）。查法：
  ```ts
  const admin = await db.user.findFirst({ where: { role: "ADMIN" }, select: { id: true } });
  // 写卡时 createdById: admin.id
  ```
- **写进库的卡，`verificationStatus` 一律 `NEEDS_REVIEW`**。执行者没有权力盖"已核实"章，升 VERIFIED 是作者审核后的手动动作。
- **Windows 环境坑**（本项目在 Windows 跑）：
  - shell 是 bash，但 cwd 可能在命令间被重置回根目录——脚本里用绝对路径，或每条命令前 `cd /d/nju-forum`。
  - 用 `npx.cmd` / `npm.cmd`，不是 `npx` / `npm`。
  - **跑 `prisma migrate` 前必须先停 dev server**，否则 DLL 被占会 EPERM 锁库报错。
  - TS 字符串内容里**不要用 ASCII 双引号**做强调（会截断字符串），用直角引号「」或不加。
- **临时脚本用完即删**，尤其涉及任何 key 的探针脚本，别留在磁盘。
