# 知识卡全量复查与修复执行计划

## 1. 文档目的

这份文档只解决一件事：

> 当知识卡已经入库，但历史修复错误较多时，如何在 **不联网、只用本地材料** 的前提下，对全库卡片做一轮可重复、可交接、可追踪的复查和修复。

这不是新增制卡流程，也不是产品设计文档。

- 单张卡怎么写，优先看 [CARD-WORKFLOW.md](./CARD-WORKFLOW.md)
- 人机协作和工作流迭代，优先看 [卡片工作流-使用与迭代.md](./卡片工作流-使用与迭代.md)

这份文档面向的是：

- 需要在 `D:\nju-forum` 内继续修卡的 agent
- 需要接手历史错卡治理的人
- 需要知道“先查什么、后改什么、改完怎么验”的执行者

---

## 2. 本轮任务边界

### 2.1 要做什么

对当前知识卡做一轮全量复查，并按本地证据修复以下问题：

1. 标题和正文错配
2. 正文方向对，但遗漏来源里的关键信息
3. `sourceExcerpt` 不能支撑正文
4. 原始申卡正确，但后续导入、覆盖或合并弄错
5. 同主题重复卡或近重复卡

### 2.2 不做什么

1. 不联网
2. 不处理“缺结果卡”
3. 不安装依赖
4. 不做 Git 写操作
5. 不启动长期服务
6. 不修改项目外文件
7. 不凭常识补写来源中没有的事实

### 2.3 本轮默认约束

- 只以本地材料为准
- 所有写库操作都必须先备份
- 所有修复都必须能回溯到本地 `cards.json`、`gpt-review.md`、`iteration.md`、PDF 或当前数据库字段
- 如果本地证据不够，宁可收缩正文，也不要补写

---

## 3. 执行总原则

### 原则 A：先审计，再修复

不要直接批量改库。必须先把问题卡分型，确认是：

- 原始申卡就有问题
- 原始申卡没问题，但后续导入错了
- 我方之前修了一半
- 当前正文超出了现有 `sourceExcerpt`

### 原则 B：按来源批次修，不按随机卡号修

优先以同一来源或同一工作流任务为单位处理，例如：

- 学生邮箱
- 校园网
- 转专业指南
- 光电

原因：同一来源下的卡最容易互相串内容，放在一起比逐张散修更稳。

### 原则 C：证据优先

正文、`sourceExcerpt`、标题三者必须互相对齐。

遇到冲突时，优先级如下：

1. 本地原始来源材料
2. 本地工作流产物（`cards.json` / review / iteration）
3. 当前数据库内容
4. agent 推断

### 原则 D：高风险问题优先

优先修：

1. 标题问 A、正文答 B
2. 当前网页展示会直接误导用户的卡
3. 原始申卡正确、后续导入覆盖错配的卡

## 4. 本轮问题分型

复查时把每张卡归到以下一种主类型。

### A 类：标题-正文错配

表现：

- 标题问的是 A
- 正文回答的是 B

处理：

- 优先保留标题
- 若本地证据能支撑，则重写正文回到标题问题
- 只有在标题本身就是错的情况下才改标题

### B 类：正文遗漏来源信息

表现：

- 方向基本对
- 但漏掉来源里的关键条件、范围、步骤、例外、时间点

处理：

- 补正文
- 不必改标题
- 如果遗漏点来自当前 `sourceExcerpt`，直接补

### C 类：`sourceExcerpt` 不足以支撑正文

表现：

- 正文里有内容
- 当前 `sourceExcerpt` 没有对应证据

处理分两种：

1. 本地其他材料能补齐证据  
   - 补 `sourceExcerpt`
   - 必要时同步微调正文

2. 本地材料也补不齐  
   - 收缩正文
   - 删掉证据链不足的部分

### D 类：原始申卡正确，但后续导入/覆盖错了

表现：

- 工作流产物里是对的
- 数据库当前是错的

处理：

- 以原始申卡产物为准恢复
- 通常需要同时修 `body` 和 `sourceExcerpt`

### E 类：重复卡 / 近重复卡

表现：

- 两张或多张卡在回答同一个问题
- 或者关键信息高度重叠，只是表述不同

处理：

- 先比较信息覆盖面
- 选一张保留为主卡
- 把独有信息并入主卡
- 其余卡归档，不直接删除

### F 类：缺结果卡

表现：

- 没有足够审查结果
- 没有本地证据支撑复查结论

处理：

- 本轮跳过
- 只记录，不修

---

## 5. 推荐执行顺序

按以下顺序推进，不要倒过来。

### 第一轮：A 类 + D 类

目标：先消灭最明显的错卡。

这类卡最容易在真实页面中直接误导用户，也是用户最容易发现的问题。

### 第二轮：C 类

目标：补齐证据链。

这类卡看起来像“差不多对”，但风险在于正文和来源脱钩。

### 第三轮：B 类

目标：补全遗漏信息。

这类卡方向正确，修起来相对稳定，适合在前两轮之后集中做。

### 第四轮：E 类

目标：清理重复卡。

重复卡一定要放最后。否则前面修卡时容易误把旧卡当成正确来源继续污染。

### 第五轮：抽样总验收

按主题抽样，不全量点网页。

每个主题抽 2 到 5 张，检查：

1. 数据库内容已更新
2. 真实 UI 能显示修正结果
3. 真实问答路径能召回正确卡

## 7. 单张卡复查 SOP

每张卡都按这个顺序走。

### Step 1：读取当前数据库卡

至少读取这些字段：

- `id`
- `summary`
- `body`
- `sourceExcerpt`
- `sourceUrl`
- `sourceDescription`
- `sourceType`
- `verificationStatus`
- `domainTag`

### Step 2：定位本地证据

至少定位到以下之一：

1. 对应工作流任务下的 `cards.json`
2. `gpt-review.md`
3. `iteration.md` 或 `transcript-gpt-review-iteration.md`
4. 本地 PDF 原文
5. 之前修卡时的整卡备份

### Step 3：判断主问题类型

只能从 A/B/C/D/E/F 中选一个主类型。

如果同时命中多个，以风险更高的那类为主。

### Step 4：决定修改策略

只能选以下之一：

1. 保留不改
2. 只改 `body`
3. 改 `body + sourceExcerpt`
4. 改 `summary + body`
5. 归档重复卡

### Step 5：写入前说明

执行写库前，必须先输出一句简短说明，至少包含：

- 改哪张卡
- 改哪些字段
- 依据是什么
- 为什么这么改

### Step 6：写库

长中文字段一律优先用：

- [update-card-fields.js](D:/nju-forum/.agents/skills/fix-cards/scripts/update-card-fields.js)

不要把长中文正文直接塞进 `node -e`。

### Step 7：立即回查

写库后必须立刻读回目标卡，确认：

1. 字段已经更新
2. 标题和正文不再错配
3. `sourceExcerpt` 能支撑正文
4. 中文无乱码

## 8. 批次执行 SOP

这里的“批次”指一个来源主题或一个工作流任务，不是固定数量。

### Step 1：选一个主题批次

例如：

- `002-学生邮箱使用`
- `010-新生校园网使用指南`
- `005-转专业指南`
- `002-如果你想转光电`

### Step 2：列出该批次关联卡

来源包括：

- 工作流 `cards.json`
- 当前数据库同 `sourceUrl`
- 当前数据库同主题关键词

### Step 3：为这批卡做问题分型台账

台账最少包含：

- 卡 ID
- 当前标题
- 主问题类型
- 修复策略
- 证据文件
- 是否已改

### Step 4：按风险顺序逐张修

顺序：

1. 错配卡
2. 证据链不足卡
3. 信息遗漏卡
4. 重复卡

### Step 5：批次结束后做一次抽查

至少抽 2 张卡：

1. 一张最复杂的
2. 一张最容易被用户直接问到的

## 9. 推荐命令与工具使用

### 9.1 查询类命令

优先使用：

- `rg`
- Prisma 只读脚本
- `Get-Content`

### 9.2 写库类命令

优先使用：

- `node .agents/skills/fix-cards/scripts/update-card-fields.js`

示例：

```powershell
$env:CARD_BODY = @'
新的正文
'@
$env:CARD_EXCERPT = @'
新的来源摘要
'@
node .agents/skills/fix-cards/scripts/update-card-fields.js `
  --id <card-id> `
  --body-env CARD_BODY `
  --source-excerpt-env CARD_EXCERPT `
  --apply
```

### 9.3 数据库回查

每次写完后都要用 Prisma 再读一次目标卡，不接受“写完就算”。

### 9.4 UI 抽查

本轮不要求每张卡都走前端。

只在每个主题批次结束后抽查。

优先验证路径：

- `http://localhost:3000/knowledge`
- `/api/knowledge/ask`

如果本地服务没在跑，不要擅自启动长期服务，先停在数据库回查层。

## 10. 接受标准

一张卡只有满足以下全部条件，才算修完。

1. `summary` 与 `body` 问答一致
2. `body` 能被本地 `sourceExcerpt` 或本地来源材料支撑
3. `sourceUrl` / `sourceDescription` 不跑题
4. 回查后的数据库字段正确
5. 中文内容无乱码

一个批次只有满足以下全部条件，才算完成。

1. 批次内卡已全部分型
2. 高风险卡已全部处理
3. 跳过项已单独记录
4. 至少做过一次真实路径抽查或明确记录“本轮未做 UI 抽查”

---

## 11. 遇到以下情况必须停下，不要硬修

1. 当前正文和本地证据都支撑不了标题
2. 当前 `sourceExcerpt` 明显截断，但本地找不到可恢复材料
3. 同主题多卡互相污染，无法判断谁是主卡
4. 当前数据库内容和工作流内容都可能是错的
5. 需要联网才能判断事实对错

处理方式：

- 不写库
- 记录为阻塞项
- 明确写出缺什么证据

## 14. 当前推荐起手顺序

如果接手人不知道从哪批开始，就按这个顺序：

1. 学生邮箱
2. 校园网
3. 转专业指南
4. 光电
5. 其他高反馈主题
6. 重复卡清理

原因：

- 这几类历史错配已经被实际发现过
- 本地证据相对完整
- 用户感知最强

## 15. 一句话执行摘要

> 先按来源批次做全量分型，再优先修标题正文错配和证据链断裂卡；所有修改只用本地材料，逐张写库、逐张回查，最后按主题抽样验证真实展示。
